Filter, Interceptor, AOP를 구분하는 기준

인증, 로깅, 권한, 트랜잭션 같은 공통 관심사는 어디에 두느냐에 따라 코드의 결합도가 달라집니다. 실행 위치를 기준으로 나누면 훨씬 덜 헷갈립니다.

요청 흐름에서의 위치

셋은 모두 공통 관심사를 처리할 수 있지만 실행 위치가 다릅니다. Filter는 Servlet 영역, Interceptor는 Spring MVC 영역, AOP는 Spring Bean 메서드 호출 주변에서 동작합니다.

Client
  → Filter
  → DispatcherServlet
  → Interceptor
  → Controller
  → Service Bean method
  → AOP advice

위치가 다르기 때문에 접근할 수 있는 정보도 다릅니다. Filter는 HTTP 요청 자체에 가깝고, Interceptor는 어떤 Handler가 실행될지 알 수 있으며, AOP는 비즈니스 메서드의 인자와 반환값에 가깝습니다.

Filter: Spring MVC 앞단의 요청 처리

Filter는 Servlet Filter입니다. Spring MVC에 들어가기 전 요청과 응답을 다룰 수 있습니다. CORS, encoding, 보안 필터 체인, request logging처럼 HTTP 레벨 처리가 어울립니다.

class RequestIdFilter extends OncePerRequestFilter {
    @Override
    protected void doFilterInternal(
        HttpServletRequest request,
        HttpServletResponse response,
        FilterChain filterChain
    ) throws ServletException, IOException {
        filterChain.doFilter(request, response);
    }
}

Interceptor: Controller 호출 전후

Interceptor는 HandlerMapping 이후에 동작합니다. 어떤 Controller 메서드가 호출될지 아는 상태에서 전후 처리를 할 수 있습니다. 로그인 확인, locale 처리, 관리자 경로 접근 확인처럼 MVC 컨텍스트가 필요한 작업에 맞습니다.

class LoginInterceptor implements HandlerInterceptor {
    @Override
    public boolean preHandle(
        HttpServletRequest request,
        HttpServletResponse response,
        Object handler
    ) {
        return true;
    }
}

AOP: Bean 메서드 주변의 공통 로직

AOP는 Spring Bean 메서드 호출을 감쌉니다. 트랜잭션, 메서드 실행 시간 측정, 권한 annotation 처리처럼 비즈니스 계층의 메서드 단위 관심사에 잘 맞습니다.

@Around("@annotation(Measured)")
public Object measure(ProceedingJoinPoint joinPoint) throws Throwable {
    long start = System.currentTimeMillis();
    try {
        return joinPoint.proceed();
    } finally {
        log.info("elapsed={}", System.currentTimeMillis() - start);
    }
}

단, Spring AOP는 프록시 기반이라 같은 클래스 내부 메서드 호출에는 advice가 적용되지 않는 self-invocation 문제가 있습니다.

선택 기준

  • HTTP 요청 자체를 다루면 Filter
  • Controller 매핑 이후의 요청 흐름을 다루면 Interceptor
  • Service 메서드 실행 전후의 관심사라면 AOP
  • 트랜잭션처럼 프록시 기반 기능은 public 메서드와 외부 호출 경계를 확인
