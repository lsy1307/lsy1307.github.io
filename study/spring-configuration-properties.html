Profile과 Configuration Properties 설계

운영 실수는 코드보다 설정에서 자주 시작됩니다. 환경별 값을 분리하고 타입이 있는 설정 객체로 묶으면 실수를 줄일 수 있습니다.

Profile은 환경별 Bean과 설정을 나누는 기준

local, dev, prod, loadtest처럼 실행 환경마다 DB 주소, 로그 레벨, 외부 API endpoint가 달라질 수 있습니다. Spring Profile은 이런 차이를 분리하는 장치입니다.

spring:
  profiles:
    active: local

---
spring:
  config:
    activate:
      on-profile: prod

Profile은 편하지만 너무 많은 조건부 Bean을 만들면 실행 환경별 차이를 추적하기 어려워집니다. 가능한 값 차이는 설정으로, 실제 구현 차이가 필요한 경우에만 Profile Bean을 둡니다.

@ConfigurationProperties로 묶기

@Value를 여러 곳에 흩뿌리면 설정의 전체 구조가 보이지 않습니다. 관련 설정은 하나의 properties 클래스로 묶는 편이 읽기 쉽습니다.

@ConfigurationProperties(prefix = "storage.s3")
public record S3Properties(
    String bucket,
    String region,
    Duration presignedUrlTtl
) {
}

이렇게 하면 어떤 설정이 필요한지 타입으로 드러나고, 테스트에서도 설정 객체를 직접 만들어 넣기 쉽습니다.

secret은 코드와 분리

DB password, API key, JWT secret 같은 값은 저장소에 올리지 않습니다. 환경 변수, Parameter Store, Secrets Manager 같은 외부 저장소에서 주입합니다.

spring:
  datasource:
    url: ${DB_URL}
    username: ${DB_USERNAME}
    password: ${DB_PASSWORD}

설정 파일에는 “어떤 값이 필요하다”는 구조만 남기고 실제 값은 배포 환경에서 관리하는 쪽이 안전합니다.

설정도 시작 시점에 검증

잘못된 설정은 요청이 들어온 뒤 터지기보다 애플리케이션 시작 시점에 실패하는 편이 낫습니다. Bean Validation을 properties 클래스에 붙여 필수 값을 검증할 수 있습니다.

@Validated
@ConfigurationProperties(prefix = "jwt")
public record JwtProperties(
    @NotBlank String secret,
    @Positive Duration accessTokenTtl
) {
}
기준운영 설정은 “편하게 읽히는가”보다 “잘못 넣었을 때 빨리 실패하는가”가 중요합니다. 설정 실수는 조용히 지나가면 장애가 됩니다.
