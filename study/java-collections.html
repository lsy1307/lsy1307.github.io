List, Set, Map을 고르는 기준

컬렉션은 익숙한 구현체를 고르는 문제가 아니라, 순서·중복·조회 방식 중 무엇이 중요한지 결정하는 문제입니다.

먼저 질문할 것

컬렉션을 고르기 전에 세 가지를 먼저 봅니다. 순서가 중요한가, 중복을 허용할 것인가, 특정 key와 값을 연결해야 하는가. 그다음 데이터 크기, 조회·삽입 패턴, 동시성 요구에 맞춰 ArrayList, HashSet, LinkedHashMap 같은 구현체를 고릅니다.

  • 순서대로 처리해야 한다면 List
  • 중복을 막아야 한다면 Set
  • key로 값을 찾는다면 Map

List: 순서와 인덱스가 중요할 때

ArrayList는 가장 흔하게 쓰는 List 구현체입니다. 데이터가 들어온 순서를 유지하고, 인덱스로 접근할 수 있습니다. API 응답에서 화면에 보여줄 순서를 유지하거나, 정렬된 결과를 그대로 넘길 때 적합합니다.

List<String> names = new ArrayList<>();
names.add("java");
names.add("spring");

String first = names.get(0);

다만 특정 값이 포함되어 있는지 자주 확인해야 한다면 List.contains는 최악의 경우 전체를 훑는 O(n) 연산입니다. 순서와 중복이 필요하지 않다면 평균 O(1) 조회를 기대할 수 있는 HashSet을 검토합니다. 실제 비용은 hash 분포와 equals/hashCode 구현에 따라 달라집니다.

Set: 중복 제거와 포함 여부 확인

HashSet은 중복을 허용하지 않고, 평균적으로 빠른 포함 여부 확인을 제공합니다. 예를 들어 이미 처리한 ID인지 확인하거나 중복 태그를 제거할 때 사용할 수 있습니다. 반복 순서는 보장하지 않으므로 입력 순서가 필요하면 LinkedHashSet, 정렬 순서가 필요하면 TreeSet을 검토합니다.

Set<Long> processedIds = new HashSet<>();

if (processedIds.add(memberId)) {
    // 처음 보는 memberId일 때만 처리
}

Set이 제대로 동작하려면 객체의 equals, hashCode 기준이 중요합니다. 특히 직접 만든 객체를 Set에 넣는다면 동등성 기준을 점검해야 합니다.

Map: key 기반 조회

Map은 key와 value를 함께 저장합니다. DB에서 가져온 결과를 ID 기준으로 묶어두고 이후 로직에서 찾아야 할 때 자주 씁니다. HashMap 조회는 평균 O(1)이지만 항상 일정한 시간을 보장한다는 뜻은 아니며, 정렬이 필요하면 TreeMap처럼 다른 구현체의 비용을 선택합니다.

Map<Long, Member> memberById = members.stream()
    .collect(Collectors.toMap(Member::getId, member -> member));

Member writer = memberById.get(writerId);

같은 key가 두 번 들어올 수 있다면 병합 규칙을 정해야 합니다. 이 규칙 없이 toMap을 쓰면 중복 key에서 예외가 날 수 있습니다.

Stream은 반복문 대체가 아니라 데이터 흐름 표현

Stream은 컬렉션을 필터링하고 변환하고 모으는 흐름을 잘 드러냅니다. 다만 중간에 외부 상태를 바꾸거나 예외 처리가 복잡해진다면 평범한 반복문이 더 읽기 쉬울 수 있습니다.

List<String> activeNames = members.stream()
    .filter(Member::isActive)
    .map(Member::getName)
    .toList();
기준Stream을 썼을 때 “무엇을 걸러서 무엇으로 바꾸는지”가 바로 보이면 좋고, 디버깅 흐름이 꼬이면 반복문으로 내리는 편이 낫습니다. Java 16부터 Stream.toList()가 반환하는 목록은 수정할 수 없으므로, 가변 목록이 필요하면 collect(Collectors.toCollection(ArrayList::new))처럼 의도를 명시합니다.
