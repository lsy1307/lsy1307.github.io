Spring basic · IoC DI

Spring IoC와 DI가 하는 일

Spring을 쓰면 객체를 직접 조립하는 코드가 줄어듭니다. 핵심은 객체 생성과 연결 책임을 애플리케이션 코드 밖으로 옮기는 것입니다.

IoC: 제어의 흐름을 뒤집기

일반적인 Java 코드에서는 필요한 객체를 직접 생성합니다. 하지만 Spring에서는 객체 생성과 의존성 연결을 Spring Container가 담당합니다. 이처럼 객체를 언제 만들고 어떻게 연결할지의 제어권이 프레임워크로 넘어가는 것을 IoC라고 부릅니다.

// 직접 조립
MemberRepository repository = new MemberRepository();
MemberService service = new MemberService(repository);

// Spring 사용
@Service
class MemberService {
    private final MemberRepository repository;
}

Bean: Spring이 관리하는 객체

Spring Container가 생성하고 관리하는 객체를 Bean이라고 합니다. @Component, @Service, @Repository, @Controller 같은 어노테이션을 붙이면 컴포넌트 스캔을 통해 Bean으로 등록될 수 있습니다.

  • @Controller: HTTP 요청을 받는 계층
  • @Service: 비즈니스 흐름을 처리하는 계층
  • @Repository: 데이터 접근 계층
  • @Configuration + @Bean: 직접 생성 로직이 필요한 Bean 등록

DI: 필요한 객체를 밖에서 넣어주기

DI는 Dependency Injection, 즉 의존성 주입입니다. 클래스가 필요한 객체를 직접 만들지 않고, 생성자나 메서드를 통해 외부에서 전달받습니다.

@Service
class OrderService {
    private final OrderRepository orderRepository;

    public OrderService(OrderRepository orderRepository) {
        this.orderRepository = orderRepository;
    }
}

이렇게 하면 OrderService는 OrderRepository 구현체를 직접 만들지 않아도 됩니다. 저장소 구현을 바꾸거나 테스트 대역을 넣을 때 변경 범위가 줄어듭니다.

생성자 주입을 기본으로 두는 이유

생성자 주입은 객체가 생성될 때 필요한 의존성이 반드시 들어오도록 만듭니다. 필드 주입보다 테스트하기 쉽고, 의존성이 빠졌을 때 애플리케이션 시작 시점에 빨리 실패합니다.

기준필수 의존성은 생성자 주입을 사용합니다. 선택적인 값이나 설정은 상황에 따라 메서드 주입, 설정 프로퍼티 등을 사용할 수 있지만, 서비스의 핵심 협력 객체는 생성자에서 드러나는 편이 읽기 좋습니다.

테스트가 쉬워지는 구조

DI를 사용하면 테스트에서 가짜 구현체를 넣어 서비스 로직만 확인할 수 있습니다. 즉, 객체 생성 방식이 테스트 가능성과 직접 연결됩니다.

OrderRepository fakeRepository = new FakeOrderRepository();
OrderService orderService = new OrderService(fakeRepository);

Spring이 편해서 DI를 쓰는 것이 아니라, 의존성 방향을 명확히 해서 변경과 테스트 비용을 줄이기 위해 DI를 쓴다고 보는 편이 좋습니다.
