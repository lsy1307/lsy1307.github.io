Spring 테스트 슬라이스와 통합 테스트 기준

테스트는 많을수록 좋은 것이 아니라, 실패했을 때 원인을 좁혀주는 방식으로 나뉘어야 합니다. 빠른 테스트와 실제 환경에 가까운 테스트의 역할이 다릅니다.

UPDATED 2026.07SPRING 6.2+BOOT 3.4+
CONTENTS단위 테스트WebMvcTestDataJpaTestSpringBootTest전략

단위 테스트: Spring 없이 빠르게

순수 도메인 로직과 서비스 계산은 Spring Context 없이 테스트하는 편이 빠릅니다. 의존성이 있다면 fake나 mock을 직접 넣습니다.

class PriceCalculatorTest {
    @Test
    void discount() {
        PriceCalculator calculator = new PriceCalculator();
        assertThat(calculator.discount(10000)).isEqualTo(9000);
    }
}

단위 테스트는 실패 원인이 좁고 실행 속도가 빠릅니다. 가능한 많은 비즈니스 규칙은 이 레벨에서 검증하는 편이 좋습니다.

@WebMvcTest: Controller 계층

@WebMvcTest는 전체 애플리케이션 대신 MVC 관련 구성과 지정한 Controller를 중심으로 Context를 올립니다. 필요한 Service 협력 객체는 test context의 mock으로 대체할 수 있습니다. Spring Security가 classpath에 있으면 보안 filter도 테스트에 관여할 수 있으므로 인증·CSRF 요구를 실제 정책대로 검증합니다.

@WebMvcTest(MemberController.class)
class MemberControllerTest {
    @Autowired
    MockMvc mockMvc;

    @MockitoBean
    MemberService memberService;
}

@MockitoBean은 Spring Framework 6.2에 추가된 Bean override annotation입니다. Spring Boot의 @MockBean은 3.4부터 deprecated이며 4.0 제거 대상으로 안내되었습니다. Boot 3.3 이하를 유지하는 프로젝트라면 해당 버전에서는 @MockBean을 사용하고, annotation 이름만 섞어 쓰지 않습니다.

request body validation, status code, response JSON 구조처럼 HTTP 계약을 확인하기 좋습니다.

@DataJpaTest: Repository와 mapping

@DataJpaTest는 JPA 관련 Bean을 중심으로 테스트합니다. Entity mapping, query method, JPQL, custom repository가 의도대로 동작하는지 확인할 수 있습니다. 기본적으로 각 테스트를 transaction으로 감싸 rollback하지만, 실제 commit 시점 제약조건까지 확인해야 한다면 명시적으로 flush하거나 별도 통합 테스트를 둡니다.

@DataJpaTest
class MemberRepositoryTest {
    @Autowired MemberRepository memberRepository;
}

테스트 DB 자동 설정은 Spring Boot 버전과 datasource 구성에 따라 달라질 수 있습니다. 운영 DB 방언, locking, native query가 중요하면 embedded DB 결과만 믿지 말고 Testcontainers와 @ServiceConnection 또는 명시적 datasource 설정으로 같은 종류의 DB를 사용합니다.

@SpringBootTest: 전체 Context

@SpringBootTest는 애플리케이션 Context를 넓게 올립니다. 기본 webEnvironment = MOCK는 실제 서버를 열지 않습니다. 실제 port까지 필요한 경우 RANDOM_PORT를, servlet 환경에서 MockMvc를 쓸 경우 @AutoConfigureMockMvc를 함께 선택합니다.

Context를 넓게 올린다고 외부 시스템까지 자동으로 실제와 같아지는 것은 아닙니다. DB·메시지 브로커·외부 API를 실제 대역, container, mock 중 무엇으로 검증하는지 테스트 이름과 설정에 드러냅니다.

대신 느립니다. 모든 테스트를 통합 테스트로 만들면 피드백이 늦어지고 실패 원인도 넓어집니다.

테스트 선택 전략

  • 계산과 정책은 단위 테스트
  • HTTP 계약은 @WebMvcTest
  • 쿼리와 mapping은 @DataJpaTest
  • 주요 사용자 시나리오는 @SpringBootTest
기준annotation 이름만 보고 테스트 범위를 단정하지 말고 실제로 로드된 Bean, DB, security filter와 외부 의존성을 확인합니다. 빠른 테스트는 규칙과 계약에, 느린 통합 테스트는 핵심 경로와 wiring에 집중합니다.
