Spring 테스트 슬라이스와 통합 테스트 기준

테스트는 많을수록 좋은 것이 아니라, 실패했을 때 원인을 좁혀주는 방식으로 나뉘어야 합니다. 빠른 테스트와 실제 환경에 가까운 테스트의 역할이 다릅니다.

단위 테스트: Spring 없이 빠르게

순수 도메인 로직과 서비스 계산은 Spring Context 없이 테스트하는 편이 빠릅니다. 의존성이 있다면 fake나 mock을 직접 넣습니다.

class PriceCalculatorTest {
    @Test
    void discount() {
        PriceCalculator calculator = new PriceCalculator();
        assertThat(calculator.discount(10000)).isEqualTo(9000);
    }
}

단위 테스트는 실패 원인이 좁고 실행 속도가 빠릅니다. 가능한 많은 비즈니스 규칙은 이 레벨에서 검증하는 편이 좋습니다.

@WebMvcTest: Controller 계층

@WebMvcTest는 MVC 관련 Bean만 올려 Controller 요청/응답을 검증합니다. Service는 mock으로 대체하는 경우가 많습니다.

@WebMvcTest(MemberController.class)
class MemberControllerTest {
    @Autowired MockMvc mockMvc;
    @MockBean MemberService memberService;
}

request body validation, status code, response json 구조처럼 HTTP 계약을 확인하기 좋습니다.

@DataJpaTest: Repository와 mapping

@DataJpaTest는 JPA 관련 Bean을 중심으로 테스트합니다. Entity mapping, query method, JPQL, custom repository가 의도대로 동작하는지 확인할 수 있습니다.

@DataJpaTest
class MemberRepositoryTest {
    @Autowired MemberRepository memberRepository;
}

실제 운영 DB와 방언 차이가 중요한 쿼리라면 H2만 믿기보다 Testcontainers로 MySQL/PostgreSQL을 띄우는 방식을 고려합니다.

@SpringBootTest: 전체 Context

@SpringBootTest는 애플리케이션 Context를 넓게 올립니다. 설정, Bean wiring, 트랜잭션, 외부 연동 mock까지 포함해 실제 흐름에 가깝게 검증할 수 있습니다.

대신 느립니다. 모든 테스트를 통합 테스트로 만들면 피드백이 늦어지고 실패 원인도 넓어집니다.

테스트 선택 전략

  • 계산과 정책은 단위 테스트
  • HTTP 계약은 @WebMvcTest
  • 쿼리와 mapping은 @DataJpaTest
  • 주요 사용자 시나리오는 @SpringBootTest
기준테스트가 실패했을 때 “어느 층이 깨졌는지” 바로 좁혀지면 좋은 테스트 구조입니다. 빠른 테스트를 많이 두고, 느린 통합 테스트는 핵심 경로에 집중합니다.
