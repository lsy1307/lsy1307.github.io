Java 객체, 참조, equals/hashCode 기본

Java에서 변수는 값 자체를 들고 있을 때도 있고, 객체가 있는 위치를 가리킬 때도 있습니다. 비교 코드를 읽을 때 이 차이를 먼저 잡아야 합니다.

값 타입과 참조 타입

int, long, boolean 같은 primitive type 변수는 실제 값을 저장합니다. 반면 String, User, List 같은 reference type 변수는 객체 자체가 아니라 객체를 가리키는 참조를 저장합니다.

int a = 10;
int b = a;       // 값 10이 복사됨

User u1 = new User("kim");
User u2 = u1;    // 같은 User 객체를 가리키는 참조가 복사됨

이 차이 때문에 객체를 메서드로 넘기면 “객체가 복사된다”기보다 “같은 객체를 가리키는 참조 값이 복사된다”고 이해하는 편이 안전합니다. 메서드 안에서 객체의 필드를 바꾸면 바깥에서도 그 변경이 보일 수 있습니다.

==와 equals

==는 primitive type에서는 값을 비교하지만, reference type에서는 두 변수가 같은 객체를 가리키는지 비교합니다. 객체의 내용이 같은지를 비교하려면 보통 equals를 사용합니다.

String s1 = new String("spring");
String s2 = new String("spring");

System.out.println(s1 == s2);      // false: 서로 다른 객체
System.out.println(s1.equals(s2)); // true: 문자열 내용이 같음

도메인 객체도 마찬가지입니다. 회원 ID가 같으면 같은 회원으로 볼 것인지, 모든 필드가 같아야 같은 값 객체로 볼 것인지 기준을 먼저 정해야 합니다. 특히 DB가 ID를 생성하는 JPA 엔티티는 저장 전 ID가 null이므로, Objects.equals(id, other.id)만 사용하면 저장 전의 서로 다른 두 엔티티를 같다고 판단하는 오류가 생깁니다.

실무 기준값 객체는 불변인 의미 필드를 기준으로 비교하기 쉽습니다. 반면 JPA 엔티티는 ID 생성 시점과 Hibernate 프록시까지 고려해야 하므로, 식별자 정책을 정하지 않은 채 IDE가 만든 equals/hashCode를 그대로 쓰지 않습니다.

hashCode까지 같이 봐야 하는 이유

HashSet, HashMap은 내부적으로 hashCode를 먼저 사용하고, 필요할 때 equals로 다시 확인합니다. 그래서 equals를 재정의했다면 hashCode도 같은 기준으로 재정의해야 합니다.

import org.hibernate.Hibernate;

class Member {
    private Long id; // DB가 생성하는 식별자라고 가정

    @Override
    public final boolean equals(Object o) {
        if (this == o) return true;
        if (o == null || Hibernate.getClass(this) != Hibernate.getClass(o)) {
            return false;
        }
        Member member = (Member) o;
        return id != null && id.equals(member.id);
    }

    @Override
    public final int hashCode() {
        return Hibernate.getClass(this).hashCode();
    }
}

이 예시는 Hibernate와 DB 생성 ID를 사용하는 경우의 한 전략입니다. ID가 null이면 동일 인스턴스가 아닌 두 엔티티는 같지 않고, 저장 후 ID가 생겨도 hash 값은 바뀌지 않아 HashSet에서 객체를 잃어버리는 문제를 피합니다. 불변 자연키를 식별자로 쓰거나 다른 JPA provider를 사용한다면 그 조건에 맞는 전략을 선택해야 합니다.

두 객체가 equals로 같다면 hashCode도 같아야 합니다. 반대로 hashCode가 같다고 항상 equals가 true인 것은 아닙니다. 해시 충돌이 있을 수 있기 때문입니다.

정리

  • primitive type은 값을 들고, reference type은 객체를 가리키는 참조를 듭니다.
  • ==는 객체의 내용 비교가 아니라 참조 비교입니다.
  • 객체의 동등성 기준은 equals와 hashCode 계약을 함께 만족해야 합니다.
  • DB 생성 ID 엔티티는 null ID와 저장 전후 hash 안정성, 프록시 타입을 함께 점검합니다.
