Generics와 Optional을 안전하게 쓰는 법

Generics는 타입 실수를 컴파일 시점에 잡기 위한 장치이고, Optional은 값이 없을 수 있음을 반환 타입에 드러내는 장치입니다.

Generics: 타입 정보를 코드에 남기기

Generics를 쓰면 컬렉션이나 공통 응답 객체가 어떤 타입을 담는지 명확해집니다. 타입 캐스팅을 줄이고, 잘못된 타입을 넣는 실수를 컴파일 시점에 잡을 수 있습니다.

List<String> names = new ArrayList<>();
names.add("spring");

String first = names.get(0); // 별도 casting 필요 없음

API 응답 래퍼처럼 여러 타입을 담아야 하는 구조에서도 Generics가 유용합니다.

record ApiResponse<T>(
    String code,
    T data
) {
}

wildcard는 읽기와 쓰기 방향을 나눌 때 사용

? extends T는 T의 하위 타입을 읽는 생산자에 가깝고, ? super T는 T를 넣는 소비자에 가깝습니다. 자주 외우는 문장으로는 PECS가 있습니다. Producer Extends, Consumer Super입니다.

void printAll(List<? extends Number> numbers) {
    for (Number number : numbers) {
        System.out.println(number);
    }
}

대부분의 서비스 코드에서는 복잡한 wildcard보다 명확한 타입 파라미터가 더 읽기 좋습니다. 라이브러리성 코드나 범용 유틸리티를 만들 때 wildcard의 가치가 커집니다.

Optional: 없을 수 있음을 반환 타입으로 표현

Optional은 null일 수 있는 반환값을 명시적으로 표현합니다. Repository의 단건 조회 결과처럼 “없음”이 자연스러운 경우에 잘 맞습니다.

Optional<Member> findByEmail(String email);

Member member = memberRepository.findByEmail(email)
    .orElseThrow(() -> new MemberNotFoundException(email));

Optional을 사용하면 호출자가 값이 없을 가능성을 무시하기 어려워집니다. null check가 사라진다기보다, null 가능성을 타입과 메서드 체인 안으로 이동시키는 느낌에 가깝습니다.

주의할 점

  • Entity 필드에 Optional을 두지 않습니다. 필드는 실제 값이나 null로 두고, 반환 메서드에서 Optional을 고려합니다.
  • 메서드 파라미터로 Optional을 받는 것은 보통 어색합니다. 호출자가 값을 꺼내서 넘기거나 메서드를 분리하는 편이 낫습니다.
  • Optional.get()만 바로 호출하면 null check보다 나을 게 없습니다. orElseThrow, map, filter로 의도를 드러냅니다.
