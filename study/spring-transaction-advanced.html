트랜잭션 전파, readOnly, rollback 심화

트랜잭션은 붙이는 것보다 경계를 제대로 잡는 게 어렵습니다. 전파 옵션과 rollback 규칙을 모르면 로그 저장, 이벤트 발행, 외부 호출에서 예상과 다른 결과가 나올 수 있습니다.

Spring 트랜잭션은 프록시 기반

@Transactional은 일반적인 proxy mode에서 Spring AOP 프록시를 통해 동작합니다. 외부에서 Bean 메서드를 호출할 때 프록시의 transaction advisor가 트랜잭션을 시작하고, 메서드 종료 결과에 따라 commit 또는 rollback을 결정합니다. AspectJ mode를 선택하면 적용 방식이 다릅니다.

같은 클래스 안에서 this.inner()처럼 내부 메서드를 호출하면 프록시를 거치지 않아 트랜잭션 설정이 적용되지 않을 수 있습니다. 이 self-invocation은 트랜잭션이 “붙어 있는데 안 먹는” 대표적인 원인입니다.

Propagation.REQUIRED는 기존 트랜잭션에 참여

기본 전파 옵션인 REQUIRED는 이미 트랜잭션이 있으면 참여하고, 없으면 새로 만듭니다. 대부분의 서비스 작업에는 이 기본값이 맞습니다.

@Transactional
public void placeOrder() {
    orderService.create();
    paymentService.pay();
}

같은 물리 트랜잭션에 참여한 proxied 작업이 예외를 밖으로 전파해 rollback 규칙을 만족하면 전체 작업이 rollback됩니다. 직접 호출한 일반 Java 메서드나 메서드 안에서 잡아버린 예외까지 프레임워크가 자동으로 알아내는 것은 아닙니다.

REQUIRES_NEW는 독립 트랜잭션

REQUIRES_NEW는 기존 트랜잭션을 중단해 둔 채 별도 물리 트랜잭션을 시작합니다. 실패 감사 기록처럼 바깥 작업과 commit 운명을 분리해야 할 때 고려할 수 있습니다. 아래 메서드는 다른 Bean을 통해 호출해야 proxy를 거치며, 같은 클래스의 self-invocation이면 전파 설정이 적용되지 않을 수 있습니다.

@Transactional(propagation = Propagation.REQUIRES_NEW)
public void saveFailureLog(String reason) {
    failureLogRepository.save(new FailureLog(reason));
}

하지만 남용하면 전체 일관성이 깨질 수 있습니다. 바깥 트랜잭션은 실패했는데 안쪽 데이터만 commit되는 결과가 의도인지 확인해야 합니다. 바깥 트랜잭션이 connection을 잡은 채 새 connection을 요구할 수 있으므로 동시 요청 수에 비해 connection pool이 작으면 고갈이나 교착 위험도 커집니다.

readOnly는 읽기 의도를 명시한다

@Transactional(readOnly = true)는 읽기 의도를 transaction manager와 provider에 전달하는 hint입니다. JPA에서는 flush mode 최적화에 도움을 줄 수 있고 일부 DB·driver는 읽기 전용 최적화를 적용할 수 있지만, 모든 환경에서 쓰기를 물리적으로 막아주는 보안 장치로 보면 안 됩니다. 조회 서비스에는 readOnly를 기본으로 두고 상태 변경 메서드에서 제거하는 기준을 세울 수 있습니다.

@Transactional(readOnly = true)
public MemberResponse getMember(Long id) {
    Member member = memberRepository.findById(id).orElseThrow();
    return MemberResponse.from(member);
}

rollback-only 함정

별도 Bean의 내부 REQUIRED 메서드가 같은 물리 트랜잭션에 참여했다가 runtime exception으로 rollback-only를 표시한 뒤, 바깥 메서드가 그 예외를 잡고 정상 종료하려 하면 최종 commit 시점에 UnexpectedRollbackException이 날 수 있습니다.

기본 규칙은 RuntimeException과 Error rollback, checked exception commit입니다. checked exception도 rollback해야 한다면 rollbackFor를 명시하고, 반대로 noRollbackFor를 쓸 때는 부분 commit이 의도인지 테스트합니다.

기준트랜잭션 안에서 예외를 삼키는 코드는 위험합니다. 실패를 복구할 수 있는지, rollback되어야 하는지, 별도 트랜잭션으로 기록해야 하는지를 먼저 정해야 합니다.
