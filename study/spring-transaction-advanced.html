트랜잭션 전파, readOnly, rollback 심화

트랜잭션은 붙이는 것보다 경계를 제대로 잡는 게 어렵습니다. 전파 옵션과 rollback 규칙을 모르면 로그 저장, 이벤트 발행, 외부 호출에서 예상과 다른 결과가 나올 수 있습니다.

Spring 트랜잭션은 프록시 기반

@Transactional은 대체로 Spring AOP 프록시를 통해 동작합니다. 외부에서 Bean 메서드를 호출할 때 프록시가 트랜잭션을 시작하고, 메서드 종료 후 commit 또는 rollback을 결정합니다.

같은 클래스 안에서 this.inner()처럼 내부 메서드를 호출하면 프록시를 거치지 않아 트랜잭션 설정이 적용되지 않을 수 있습니다. 이 self-invocation은 트랜잭션이 “붙어 있는데 안 먹는” 대표적인 원인입니다.

Propagation.REQUIRED는 기존 트랜잭션에 참여

기본 전파 옵션인 REQUIRED는 이미 트랜잭션이 있으면 참여하고, 없으면 새로 만듭니다. 대부분의 서비스 작업에는 이 기본값이 맞습니다.

@Transactional
public void placeOrder() {
    orderService.create();
    paymentService.pay();
}

내부 작업 중 하나가 unchecked exception을 던지면 전체 트랜잭션이 rollback됩니다. 이것이 “하나의 비즈니스 작업 단위”라는 의미입니다.

REQUIRES_NEW는 독립 트랜잭션

REQUIRES_NEW는 기존 트랜잭션을 잠시 멈추고 새 트랜잭션을 시작합니다. 실패한 요청의 로그를 반드시 남기거나, 외부 이벤트 발행 상태를 별도 기록해야 할 때 고려할 수 있습니다.

@Transactional(propagation = Propagation.REQUIRES_NEW)
public void saveFailureLog(String reason) {
    failureLogRepository.save(new FailureLog(reason));
}

하지만 남용하면 전체 일관성이 깨질 수 있습니다. 바깥 트랜잭션은 실패했는데 안쪽 로그나 일부 데이터만 commit되는 결과가 의도인지 확인해야 합니다.

readOnly는 읽기 의도를 명시한다

@Transactional(readOnly = true)는 읽기 작업임을 드러내고, JPA에서는 flush mode 최적화에 도움을 줄 수 있습니다. 조회 서비스에는 readOnly를 기본으로 두고, 상태 변경 메서드에서만 readOnly를 제거하는 식으로 기준을 세울 수 있습니다.

@Transactional(readOnly = true)
public MemberResponse getMember(Long id) {
    Member member = memberRepository.findById(id).orElseThrow();
    return MemberResponse.from(member);
}

rollback-only 함정

내부 트랜잭션에서 예외가 발생해 rollback-only로 표시된 뒤 바깥에서 예외를 잡고 정상 종료하려 하면, 최종 commit 시점에 UnexpectedRollbackException이 날 수 있습니다.

기준트랜잭션 안에서 예외를 삼키는 코드는 위험합니다. 실패를 복구할 수 있는지, rollback되어야 하는지, 별도 트랜잭션으로 기록해야 하는지를 먼저 정해야 합니다.
