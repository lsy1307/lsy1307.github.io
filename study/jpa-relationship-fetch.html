JPA · Fetch strategy

연관관계 매핑과 Fetch 전략

JPA의 연관관계는 객체 탐색을 편하게 만들지만, 조회 범위를 잘못 잡으면 쿼리 수와 응답 시간이 크게 늘어납니다.

UPDATED 2026.07JPA 3HIBERNATE 6
CONTENTS주인LAZYN+1fetch joinDTO 조회

연관관계의 주인은 외래키를 관리하는 쪽

양방향 연관관계에서 mappedBy가 없는 쪽이 연관관계의 주인입니다. 보통 외래키를 가진 @ManyToOne 쪽이 주인입니다.

@Entity
class Order {
    @ManyToOne(fetch = FetchType.LAZY)
    @JoinColumn(name = "member_id")
    private Member member;
}

@Entity
class Member {
    @OneToMany(mappedBy = "member")
    private List<Order> orders = new ArrayList<>();
}

DB에는 외래키가 하나뿐입니다. JPA가 DB에 외래키 변경을 반영하는 기준은 owning side이고, 양방향 객체 그래프의 양쪽 값은 애플리케이션이 맞춰야 합니다.

public void changeMember(Member member) {
    if (this.member != null) {
        this.member.getOrders().remove(this);
    }
    this.member = member;
    if (member != null && !member.getOrders().contains(this)) {
        member.getOrders().add(this);
    }
}

편의 메서드는 한쪽에만 두고 두 컬렉션이 중복되지 않도록 설계합니다. 컬렉션의 contains/remove는 엔티티의 equals/hashCode 정책과 연결되므로 저장 전 ID 처리도 함께 점검합니다.

기본은 LAZY를 우선한다

JPA 명세에서 @ManyToOne, @OneToOne의 기본 fetch는 EAGER입니다. provider가 EAGER를 언제 어떤 SQL로 처리할지는 달라질 수 있으므로, 실무에서는 to-one 관계도 LAZY로 명시하고 필요한 조회에서 fetch 전략을 따로 잡는 방식을 자주 선택합니다.

@ManyToOne(fetch = FetchType.LAZY)
private Member member;

LAZY는 무조건 좋은 것이 아니라 “필요할 때 가져온다”는 전략입니다. 반복문 안에서 LAZY 연관관계를 계속 접근하면 N+1 문제가 생길 수 있습니다.

N+1은 목록 조회 후 연관 조회가 반복되는 문제

주문 목록 100개를 조회한 뒤 각 주문의 회원에 접근하면 추가 조회가 반복될 수 있습니다. 1차 캐시나 batch fetching에 따라 정확히 100번이 아닐 수도 있지만, 결과 건수에 비례해 조회가 늘어나는 패턴인지 SQL과 통계로 확인합니다.

List<Order> orders = orderRepository.findAll();

for (Order order : orders) {
    order.getMember().getName(); // 추가 쿼리 반복 가능
}

fetch join은 필요한 그래프를 한 번에 가져온다

응답에 연관 객체가 반드시 필요하다면 fetch join으로 한 번에 가져올 수 있습니다.

@Query("""
    select o
    from Order o
    join fetch o.member
    where o.status = :status
""")
List<Order> findWithMemberByStatus(OrderStatus status);

위 예시처럼 to-one 관계를 fetch join하면 root 행이 증식하지 않아 일반적으로 pagination과 함께 쓰기 쉽습니다. 반면 to-many collection fetch join은 root 행을 중복시키므로 pagination 결과가 달라질 수 있습니다. Hibernate의 SQL 처리 방식은 버전과 query 형태에 따라 달라질 수 있어 “항상 메모리 pagination” 또는 “Hibernate 6이면 항상 안전”이라고 단정하지 않습니다.

spring:
  jpa:
    properties:
      hibernate.query.fail_on_pagination_over_collection_fetch: true

운영에서는 위 옵션으로 위험한 collection fetch pagination을 조기에 실패시키고, SQL과 warning을 확인합니다. 안정적인 대안은 먼저 root ID를 page로 조회한 뒤 두 번째 쿼리에서 필요한 연관 데이터를 가져오거나, batch fetching·DTO projection을 사용하는 것입니다. ID 목록 조회 후의 두 번째 쿼리는 첫 page의 정렬 순서를 별도로 복원해야 할 수 있습니다.

읽기 전용 화면은 DTO projection도 검토

화면에 필요한 필드가 일부라면 엔티티 그래프 전체를 가져오지 않고 DTO로 바로 조회할 수 있습니다. 조회 모델과 변경 모델을 분리하면 객체 그래프 범위를 줄일 수 있습니다.

기준도메인 로직을 실행해야 하면 엔티티 조회가 자연스럽고, 단순 화면 응답이라면 DTO projection이나 ID 기반 일괄 조회가 더 단순할 수 있습니다.
