JPA · Fetch strategy

연관관계 매핑과 Fetch 전략

JPA의 연관관계는 객체 탐색을 편하게 만들지만, 조회 범위를 잘못 잡으면 쿼리 수와 응답 시간이 크게 늘어납니다.

연관관계의 주인은 외래키를 관리하는 쪽

양방향 연관관계에서 mappedBy가 없는 쪽이 연관관계의 주인입니다. 보통 외래키를 가진 @ManyToOne 쪽이 주인입니다.

@Entity
class Order {
    @ManyToOne(fetch = FetchType.LAZY)
    @JoinColumn(name = "member_id")
    private Member member;
}

@Entity
class Member {
    @OneToMany(mappedBy = "member")
    private List<Order> orders = new ArrayList<>();
}

DB에는 외래키가 하나뿐입니다. 객체 양쪽에 값을 맞추는 편의 메서드를 두면 메모리상의 객체 그래프와 DB 반영 기준을 일치시키기 쉽습니다.

기본은 LAZY를 우선한다

@ManyToOne, @OneToOne은 기본 fetch가 EAGER라서 의도치 않은 즉시 로딩을 만들 수 있습니다. 실무에서는 대부분 LAZY로 명시하고, 필요한 조회에서 fetch 전략을 따로 잡는 편이 안전합니다.

@ManyToOne(fetch = FetchType.LAZY)
private Member member;

LAZY는 무조건 좋은 것이 아니라 “필요할 때 가져온다”는 전략입니다. 반복문 안에서 LAZY 연관관계를 계속 접근하면 N+1 문제가 생길 수 있습니다.

N+1은 목록 조회 후 연관 조회가 반복되는 문제

주문 목록 100개를 조회한 뒤 각 주문의 회원을 접근하면 회원 조회가 100번 추가될 수 있습니다. 데이터가 적을 때는 잘 안 보이다가 운영 데이터에서 응답 시간이 튀는 전형적인 패턴입니다.

List<Order> orders = orderRepository.findAll();

for (Order order : orders) {
    order.getMember().getName(); // 추가 쿼리 반복 가능
}

fetch join은 필요한 그래프를 한 번에 가져온다

응답에 연관 객체가 반드시 필요하다면 fetch join으로 한 번에 가져올 수 있습니다.

@Query("""
    select o
    from Order o
    join fetch o.member
    where o.status = :status
""")
List<Order> findWithMemberByStatus(OrderStatus status);

다만 collection fetch join과 pagination을 같이 쓰면 주의가 필요합니다. DB level pagination이 아니라 메모리 pagination으로 이어질 수 있기 때문입니다.

읽기 전용 화면은 DTO projection도 검토

화면에 필요한 필드가 일부라면 엔티티 그래프 전체를 가져오지 않고 DTO로 바로 조회할 수 있습니다. 조회 모델과 변경 모델을 분리하면 객체 그래프 범위를 줄일 수 있습니다.

기준도메인 로직을 실행해야 하면 엔티티 조회가 자연스럽고, 단순 화면 응답이라면 DTO projection이나 ID 기반 일괄 조회가 더 단순할 수 있습니다.
