Spring Security 인증·인가 흐름 기본

보안 코드는 “로그인했는가”와 “이 작업을 할 권한이 있는가”를 분리해서 보면 훨씬 선명해집니다.

인증과 인가

인증(Authentication)은 사용자가 누구인지 확인하는 과정입니다. 인가(Authorization)는 그 사용자가 특정 자원이나 기능에 접근할 권한이 있는지 확인하는 과정입니다.

  • 인증: 이 요청을 보낸 사용자가 누구인가?
  • 인가: 이 사용자가 이 API를 호출해도 되는가?

로그인은 인증의 한 형태이고, 관리자 API 접근 제한은 인가의 한 형태입니다. 둘을 섞어 생각하면 “로그인만 하면 모든 작업이 가능한” 구조가 되기 쉽습니다.

SecurityFilterChain은 요청 앞단에서 동작

Spring Security는 여러 Filter로 구성된 보안 체인을 통해 요청을 검사합니다. 인증 토큰을 읽고, 인증 객체를 만들고, 권한을 확인하는 흐름이 이 체인 안에서 일어납니다.

@Configuration
@EnableMethodSecurity
class SecurityConfig {
    @Bean
    SecurityFilterChain securityFilterChain(HttpSecurity http)
        throws Exception {
        return http
            .authorizeHttpRequests(auth -> auth
                .requestMatchers("/public/**").permitAll()
                .requestMatchers("/admin/**").hasRole("ADMIN")
                .anyRequest().authenticated()
            )
            .build();
    }
}

보안 설정은 순서와 matcher 범위가 중요합니다. 먼저 일치한 authorization rule이 적용되므로 구체 규칙을 일반 규칙보다 앞에 둡니다. hasRole("ADMIN")은 기본적으로 ROLE_ADMIN authority를 확인합니다. 저장된 권한 문자열을 정확히 비교하려면 hasAuthority("ADMIN")처럼 명시하고 두 규칙을 혼용하지 않습니다.

SecurityContext는 현재 인증 정보를 담는다

인증이 성공하면 Authentication 객체가 SecurityContext에 저장됩니다. Controller나 Service에서는 현재 사용자 정보를 사용할 수 있습니다.

Authentication authentication =
    SecurityContextHolder.getContext().getAuthentication();

anonymous authentication이 활성화된 기본 구성에서는 로그인하지 않은 요청에도 AnonymousAuthenticationToken이 들어갈 수 있습니다. 따라서 authentication != null이나 isAuthenticated()만으로 로그인 여부를 판정하지 않습니다. URL 규칙에서는 authenticated()를 사용하고, 코드에서 구분해야 한다면 AuthenticationTrustResolver 또는 principal 타입을 확인합니다.

도메인 로직이 SecurityContextHolder에 직접 강하게 의존하면 테스트와 비동기 실행이 어려워질 수 있습니다. 웹 경계에서 @AuthenticationPrincipal로 현재 사용자 ID를 얻어 Service에 명시적으로 넘기거나, 별도 port로 감싸는 방식을 고려합니다.

JWT 기반 인증에서 확인할 것

JWT access token은 서버 세션 없이 검증하는 방식으로 사용할 수 있지만, JWT를 쓴다고 시스템 전체가 자동으로 stateless가 되는 것은 아닙니다. refresh token, 강제 로그아웃, key rotation을 위해 서버 상태를 둘 수 있습니다. 중요한 것은 토큰을 단순 decode한 것이 아니라 신뢰한 algorithm과 key로 검증했다는 점입니다.

  • 서명 검증
  • 만료 시간 검증
  • issuer, audience 같은 발급 정보 검증
  • 탈취 대응을 위한 refresh token 정책

메서드 보안은 비즈니스 권한에 가깝다

URL 패턴만으로 권한을 모두 표현하기 어려울 때 메서드 보안을 사용할 수 있습니다. @PreAuthorize를 사용하려면 설정에 @EnableMethodSecurity가 필요합니다. 예를 들어 “프로젝트 멤버만 수정 가능” 같은 규칙은 URL보다 도메인 데이터 확인이 필요합니다.

@PreAuthorize("@projectGuard.canEdit(#projectId, authentication)")
public void updateProject(Long projectId, UpdateProjectRequest request) {
    // 수정 로직
}
기준인증과 URL 단위 접근 제한은 SecurityFilterChain에서 먼저 처리하고, 도메인 데이터가 필요한 권한 판단은 Service와 method security에서 방어적으로 적용합니다. 화면 숨김이나 MVC Interceptor만으로 서버 인가를 대신하지 않습니다.
