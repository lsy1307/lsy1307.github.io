Spring Security 인증·인가 흐름 기본

보안 코드는 “로그인했는가”와 “이 작업을 할 권한이 있는가”를 분리해서 보면 훨씬 선명해집니다.

인증과 인가

인증(Authentication)은 사용자가 누구인지 확인하는 과정입니다. 인가(Authorization)는 그 사용자가 특정 자원이나 기능에 접근할 권한이 있는지 확인하는 과정입니다.

  • 인증: 이 요청을 보낸 사용자가 누구인가?
  • 인가: 이 사용자가 이 API를 호출해도 되는가?

로그인은 인증의 한 형태이고, 관리자 API 접근 제한은 인가의 한 형태입니다. 둘을 섞어 생각하면 “로그인만 하면 모든 작업이 가능한” 구조가 되기 쉽습니다.

SecurityFilterChain은 요청 앞단에서 동작

Spring Security는 여러 Filter로 구성된 보안 체인을 통해 요청을 검사합니다. 인증 토큰을 읽고, 인증 객체를 만들고, 권한을 확인하는 흐름이 이 체인 안에서 일어납니다.

@Bean
SecurityFilterChain securityFilterChain(HttpSecurity http) throws Exception {
    return http
        .authorizeHttpRequests(auth -> auth
            .requestMatchers("/admin/**").hasRole("ADMIN")
            .anyRequest().authenticated()
        )
        .build();
}

보안 설정은 순서와 matcher 범위가 중요합니다. 넓은 허용 규칙이 앞에 오면 뒤의 제한 규칙이 의미 없어질 수 있습니다.

SecurityContext는 현재 인증 정보를 담는다

인증이 성공하면 Authentication 객체가 SecurityContext에 저장됩니다. Controller나 Service에서는 현재 사용자 정보를 꺼내 사용할 수 있습니다.

Authentication authentication =
    SecurityContextHolder.getContext().getAuthentication();

다만 도메인 로직이 SecurityContext에 직접 강하게 의존하면 테스트가 어려워질 수 있습니다. Controller에서 현재 사용자 ID를 꺼내 Service에 명시적으로 넘기는 방식도 자주 사용합니다.

JWT 기반 인증에서 확인할 것

JWT는 서버가 세션을 저장하지 않고 토큰 자체의 서명과 만료 시간을 검증하는 방식으로 사용할 수 있습니다. 이때 중요한 것은 토큰을 “읽었다”가 아니라 “검증했다”입니다.

  • 서명 검증
  • 만료 시간 검증
  • issuer, audience 같은 발급 정보 검증
  • 탈취 대응을 위한 refresh token 정책

메서드 보안은 비즈니스 권한에 가깝다

URL 패턴만으로 권한을 모두 표현하기 어려울 때 메서드 보안을 사용할 수 있습니다. 예를 들어 “프로젝트 멤버만 수정 가능” 같은 규칙은 URL보다 도메인 데이터 확인이 필요합니다.

@PreAuthorize("@projectGuard.canEdit(#projectId, authentication)")
public void updateProject(Long projectId, UpdateProjectRequest request) {
    // 수정 로직
}
기준URL 단위 접근 제한은 SecurityFilterChain에서, 도메인 데이터가 필요한 권한 판단은 Service 또는 method security에서 다루는 편이 자연스럽습니다.
