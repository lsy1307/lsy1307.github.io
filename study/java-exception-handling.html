예외 처리 기준: checked, unchecked, custom exception

예외 처리는 try-catch 문법보다 “어디서 복구할 수 있는가”를 정하는 문제에 가깝습니다. 이 기준이 있어야 API 응답과 트랜잭션 rollback도 흔들리지 않습니다.

STUDY NOTE6 MIN READJAVA · SPRING
CONTENTScheckeduncheckedcustomSpring rollbackAPI 응답

checked exception: 호출자가 처리하도록 강제

IOException처럼 checked exception은 컴파일러가 처리 여부를 확인합니다. 호출자가 복구 전략을 가질 수 있는 경우에는 의미가 있지만, 계층을 지나며 단순히 throws만 전파된다면 코드가 오히려 지저분해질 수 있습니다.

try {
    Files.readString(path);
} catch (IOException e) {
    throw new FileReadFailedException("파일을 읽지 못했습니다.", e);
}

외부 파일, 네트워크, 메시지 브로커처럼 실패 가능성이 정상 흐름의 일부인 작업에서는 원인을 보존하면서 애플리케이션 예외로 바꾸는 방식을 자주 씁니다.

unchecked exception: 프로그래밍 오류와 비즈니스 실패

RuntimeException 계열은 컴파일러가 처리를 강제하지 않습니다. null 접근, 잘못된 인자, 존재하지 않는 리소스, 권한 없음 같은 상황을 표현할 때 사용합니다.

Member member = memberRepository.findById(memberId)
    .orElseThrow(() -> new MemberNotFoundException(memberId));

unchecked exception을 무조건 나쁘게 볼 필요는 없습니다. 오히려 Spring 애플리케이션에서는 비즈니스 실패를 unchecked custom exception으로 표현하고, 전역 예외 처리에서 HTTP 응답으로 바꾸는 방식이 자연스럽습니다.

custom exception은 의미를 이름으로 드러낸다

IllegalArgumentException 하나로 모든 실패를 표현하면 호출부나 로그에서 이유를 빠르게 알기 어렵습니다. MemberNotFoundException, DuplicateEmailException처럼 도메인 의미가 드러나는 예외는 디버깅과 응답 매핑에 도움이 됩니다.

class DuplicateEmailException extends RuntimeException {
    DuplicateEmailException(String email) {
        super("이미 사용 중인 이메일입니다: " + email);
    }
}
기준예외 이름만 보고 “무엇이 실패했는지” 알 수 있으면 custom exception의 가치가 있습니다. 단순히 클래스 수만 늘어나는 예외라면 공통 예외와 error code로 정리하는 편이 낫습니다.

Spring rollback 기본

Spring의 @Transactional은 기본적으로 unchecked exception과 Error에서 rollback합니다. checked exception은 기본 rollback 대상이 아닙니다. 이 차이를 모르고 checked exception을 던지면 DB 변경이 commit될 수 있습니다.

@Transactional(rollbackFor = IOException.class)
public void importFile(Path path) throws IOException {
    // checked exception에서도 rollback이 필요하다면 명시
}

외부 시스템 호출과 DB 변경이 함께 있을 때는 rollback만으로 충분하지 않을 수 있습니다. DB는 되돌릴 수 있어도 이미 업로드된 파일, 발송된 메시지, 외부 API 호출은 별도 보상 로직이 필요할 수 있습니다.

API 응답은 전역 예외 처리에서 통일

Controller마다 try-catch를 두면 응답 형식이 흩어집니다. @RestControllerAdvice에서 예외를 HTTP status와 error body로 바꾸면 실패 응답을 한곳에서 관리할 수 있습니다.

@ExceptionHandler(MemberNotFoundException.class)
ResponseEntity<ErrorResponse> handle(MemberNotFoundException e) {
    return ResponseEntity.status(HttpStatus.NOT_FOUND)
        .body(new ErrorResponse("MEMBER_NOT_FOUND", e.getMessage()));
}
