JVM 메모리 구조와 GC 기본

Spring 애플리케이션의 응답 지연과 OOM을 볼 때는 JVM 메모리 구조와 GC가 어떤 비용을 만드는지 먼저 알아야 합니다.

JVM 메모리 큰 그림

JVM 메모리는 크게 Heap, Stack, Metaspace, Code Cache 등으로 나눠 볼 수 있습니다. 애플리케이션 객체 대부분은 Heap에 올라가고, 메서드 호출마다 필요한 지역 변수와 호출 정보는 각 스레드의 Stack에 쌓입니다.

JVM
  ├─ Heap: 객체가 주로 저장되는 공간
  ├─ Stack: 스레드별 메서드 호출과 지역 변수
  ├─ Metaspace: 클래스 메타데이터
  └─ Code Cache: JIT 컴파일된 코드

Stack은 스레드마다 따로 있고, Heap은 여러 스레드가 공유합니다. 그래서 공유 객체를 여러 스레드가 수정할 때 동시성 문제가 생길 수 있습니다.

Heap과 객체 생명주기

대부분의 객체는 생성 후 짧게 쓰이고 사라집니다. JVM의 세대별 GC는 이 특성을 이용해 young generation과 old generation을 나눠 관리합니다.

  • Young 영역: 새 객체가 먼저 할당되는 곳
  • Old 영역: 여러 번 GC를 버틴 객체가 이동하는 곳
  • 객체가 계속 참조되면 GC 대상이 되지 않습니다.

캐시, static 컬렉션, ThreadLocal 정리 누락처럼 참조가 계속 남는 구조는 메모리 누수로 이어질 수 있습니다.

GC는 메모리를 정리하지만 멈춤 비용을 만든다

GC는 더 이상 참조되지 않는 객체를 정리합니다. 하지만 어떤 GC든 애플리케이션 스레드를 잠시 멈추는 구간이 생길 수 있습니다. 이 시간을 Stop-The-World pause라고 부릅니다.

java -Xms512m -Xmx512m -Xlog:gc*:file=gc.log -jar app.jar

GC 로그를 보면 pause time, heap 사용량, full GC 여부를 확인할 수 있습니다. 응답 지연이 갑자기 튄다면 DB나 네트워크뿐 아니라 GC pause도 함께 봐야 합니다.

운영에서 먼저 보는 지표

  • Heap used가 GC 이후에도 계속 증가하는지
  • Full GC가 반복되는지
  • GC pause 시간이 p99 응답 시간과 겹치는지
  • 스레드 수와 Stack 메모리 사용량이 과도하지 않은지
기준메모리 문제를 볼 때는 “많이 쓴다”보다 “GC 이후에도 회수되지 않는다”가 더 중요합니다. 회수되지 않는 참조를 찾는 게 누수 분석의 시작입니다.
