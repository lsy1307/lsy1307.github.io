Lambda와 Stream을 읽기 좋게 쓰는 기준

Stream은 반복문을 짧게 쓰는 도구가 아니라 컬렉션 데이터를 어떤 흐름으로 바꾸는지 표현하는 도구입니다.

Lambda: 함수를 값처럼 넘기기

Lambda는 메서드에 동작을 넘길 수 있게 해줍니다. Java에서는 함수형 인터페이스, 즉 추상 메서드가 하나인 인터페이스 자리에 lambda를 사용할 수 있습니다.

List<Member> activeMembers = members.stream()
    .filter(member -> member.isActive())
    .toList();

lambda가 길어지면 읽기 어려워집니다. 조건이 복잡하다면 private 메서드로 이름을 붙여주는 편이 좋습니다.

members.stream()
    .filter(this::canReceiveNotification)
    .map(Member::getEmail)
    .toList();

Stream pipeline: source, intermediate, terminal

Stream은 데이터 원본에서 시작해 중간 연산을 거쳐 최종 연산으로 끝납니다. 중간 연산은 lazy하게 동작하고, 최종 연산이 실행될 때 실제 처리가 일어납니다.

List<String> result = members.stream()       // source
    .filter(Member::isActive)               // intermediate
    .map(Member::getName)                   // intermediate
    .sorted()                               // intermediate
    .toList();                              // terminal

이 흐름이 한눈에 읽히면 Stream이 좋습니다. 중간에 로그를 많이 찍어야 하거나 상태 변경이 섞이면 반복문이 더 솔직한 코드일 수 있습니다.

Collector: groupingBy, toMap

목록을 Map으로 바꾸거나 그룹핑할 때 Collector가 자주 쓰입니다. 다만 key 중복 가능성을 항상 확인해야 합니다.

Map<Long, Member> memberById = members.stream()
    .collect(Collectors.toMap(
        Member::getId,
        member -> member
    ));

Map<Team, List<Member>> membersByTeam = members.stream()
    .collect(Collectors.groupingBy(Member::getTeam));

toMap에서 같은 key가 나올 수 있다면 병합 함수를 명시합니다. “나중 값으로 덮어쓰기” 같은 정책은 코드에 드러나야 합니다.

Stream을 쓰지 않는 편이 나은 경우

  • 중간에 외부 상태를 계속 변경해야 할 때
  • break, continue처럼 흐름 제어가 핵심일 때
  • 예외 처리와 로깅이 너무 많아 pipeline이 길어질 때
  • 성능상 primitive stream이나 반복문이 더 명확한 hot path일 때
기준Stream을 썼을 때 “무엇을 걸러서 무엇으로 바꾸는지”가 더 잘 보이면 사용합니다. 코드가 멋있어 보이기만 하고 디버깅이 어려워지면 반복문이 이깁니다.
