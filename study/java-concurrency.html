Thread는 실행 흐름

Thread는 프로세스 안의 실행 흐름입니다. 여러 스레드는 번갈아 진행되는 동시성 또는 여러 코어에서 실제로 실행되는 병렬성을 만들 수 있습니다. 전통적인 servlet 기반 Spring Boot 서버는 보통 내장 서버의 요청 스레드 풀로 요청을 처리하지만, WebFlux나 virtual thread 설정처럼 실행 모델이 다른 경우도 있습니다.

스레드가 많다고 항상 빠른 것은 아닙니다. CPU보다 스레드가 과도하게 많으면 context switching 비용이 늘고, DB connection pool 같은 외부 자원에서 병목이 생길 수 있습니다.

race condition: 실행 순서에 따라 결과가 바뀐다

여러 스레드가 같은 값을 동시에 읽고 쓰면 실행 순서에 따라 결과가 달라질 수 있습니다.

class Counter {
    private int count = 0;

    void increase() {
        count++;
    }
}

count++는 한 줄이지만 실제로는 읽기, 증가, 쓰기 단계로 나뉩니다. 두 스레드가 동시에 실행하면 증가가 누락될 수 있습니다.

synchronized와 lock

synchronized는 한 번에 하나의 스레드만 임계 영역에 들어오도록 합니다. 공유 상태를 보호할 수 있지만, lock 범위가 넓으면 처리량이 줄어듭니다.

class Counter {
    private int count = 0;

    synchronized void increase() {
        count++;
    }
}

volatile은 가시성과 일부 순서 보장을 제공하지만 count++ 같은 복합 연산을 원자적으로 만들지는 않습니다. 단순 카운터라면 AtomicInteger, 분산 환경의 데이터 변경이라면 DB의 원자적 update나 적절한 잠금처럼 문제 범위에 맞는 수단을 선택합니다.

불변 객체, 지역 변수처럼 lock 없이 문제를 풀 수 있는 구조가 있다면 먼저 검토합니다. 동시성은 “잠그기”보다 “공유하지 않기”가 더 단순할 때가 많습니다.

ExecutorService: 스레드 직접 생성 대신 풀 사용

요청마다 플랫폼 스레드를 새로 만들면 비용이 큽니다. ExecutorService는 작업 실행과 스레드 관리를 분리합니다. 다만 Executors.newFixedThreadPool은 내부에 사실상 무제한 LinkedBlockingQueue를 사용하므로, 처리량보다 요청이 빠르면 대기 작업과 지연이 계속 쌓일 수 있습니다.

ThreadPoolExecutor executor = new ThreadPoolExecutor(
    4,
    4,
    0L,
    TimeUnit.MILLISECONDS,
    new ArrayBlockingQueue<>(100),
    new ThreadPoolExecutor.CallerRunsPolicy()
);

executor.execute(() -> sendEmail());

예시는 queue를 100개로 제한하고 포화 시 제출 스레드가 작업을 실행해 생산 속도를 늦춥니다. 실제 서비스에서는 허용 지연, 유실 가능성, 재시도 정책에 맞춰 queue 크기와 거절 정책을 정하고 종료 시 shutdown을 호출합니다. Spring에서는 수명주기가 관리되는 ThreadPoolTaskExecutor Bean으로 두는 편이 일반적입니다.

스레드 풀 크기는 CPU 작업인지 I/O 대기 작업인지에 따라 달라집니다. “CPU 작업은 코어 수” 같은 공식은 출발점일 뿐이며, DB connection pool·외부 API 제한·메모리와 부하테스트 결과를 함께 봐야 합니다.

Spring에서 자주 만나는 지점

  • singleton Bean은 여러 요청 스레드가 공유하므로 가변 필드를 조심합니다.
  • @Async는 별도 스레드에서 실행되므로 호출 스레드의 트랜잭션이 그대로 전파되지 않습니다. 보안·로깅 컨텍스트도 별도 전파 설정을 확인합니다.
  • DB connection pool보다 요청 스레드가 훨씬 많으면 대기 시간이 길어질 수 있습니다.
