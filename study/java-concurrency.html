Thread는 실행 흐름

프로세스 안에서 여러 작업 흐름을 동시에 실행할 때 Thread를 사용합니다. Spring Boot 서버도 요청을 처리할 때 스레드 풀의 스레드를 사용합니다.

스레드가 많다고 항상 빠른 것은 아닙니다. CPU보다 스레드가 과도하게 많으면 context switching 비용이 늘고, DB connection pool 같은 외부 자원에서 병목이 생길 수 있습니다.

race condition: 실행 순서에 따라 결과가 바뀐다

여러 스레드가 같은 값을 동시에 읽고 쓰면 실행 순서에 따라 결과가 달라질 수 있습니다.

class Counter {
    private int count = 0;

    void increase() {
        count++;
    }
}

count++는 한 줄이지만 실제로는 읽기, 증가, 쓰기 단계로 나뉩니다. 두 스레드가 동시에 실행하면 증가가 누락될 수 있습니다.

synchronized와 lock

synchronized는 한 번에 하나의 스레드만 임계 영역에 들어오도록 합니다. 공유 상태를 보호할 수 있지만, lock 범위가 넓으면 처리량이 줄어듭니다.

class Counter {
    private int count = 0;

    synchronized void increase() {
        count++;
    }
}

불변 객체, 지역 변수, DB의 원자적 update처럼 lock 없이 문제를 풀 수 있는 구조가 있다면 먼저 검토합니다. 동시성은 “잠그기”보다 “공유하지 않기”가 더 단순할 때가 많습니다.

ExecutorService: 스레드 직접 생성 대신 풀 사용

요청마다 새 스레드를 만들면 비용이 큽니다. ExecutorService는 미리 만든 스레드 풀에 작업을 제출해 재사용합니다.

ExecutorService executor = Executors.newFixedThreadPool(4);

executor.submit(() -> {
    sendEmail();
});

스레드 풀 크기는 CPU 작업인지 I/O 대기 작업인지에 따라 다르게 잡아야 합니다. CPU 중심 작업은 코어 수에 가깝게, I/O 대기가 많은 작업은 더 크게 잡을 수 있지만 외부 자원 한계를 같이 봐야 합니다.

Spring에서 자주 만나는 지점

  • singleton Bean은 여러 요청 스레드가 공유하므로 가변 필드를 조심합니다.
  • @Async는 별도 스레드에서 실행되므로 트랜잭션과 보안 컨텍스트 전파를 확인합니다.
  • DB connection pool보다 요청 스레드가 훨씬 많으면 대기 시간이 길어질 수 있습니다.
