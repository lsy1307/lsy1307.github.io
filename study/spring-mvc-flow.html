요청이 Controller까지 도착하는 흐름

Spring MVC는 HTTP 요청을 Controller 메서드에 연결해 줍니다. 이 흐름을 알면 DTO, Validation, 예외 처리 위치가 더 분명해집니다.

기본 요청 흐름

Servlet Filter를 통과한 HTTP 요청은 DispatcherServlet으로 들어옵니다. HandlerMapping이 Handler를 찾고 HandlerAdapter가 argument resolver, data binding, validation을 거쳐 Controller 메서드를 호출합니다. 반환값은 return value handler와 HttpMessageConverter 등을 거쳐 응답으로 직렬화됩니다.

Client
  → Servlet Filter chain
  → DispatcherServlet
  → HandlerMapping
  → HandlerAdapter / argument resolver
  → Controller method
  → Service
  → Repository
  → HttpMessageConverter
  → Response

개발자가 매번 이 흐름을 직접 작성하지 않아도 되는 이유는 Spring MVC가 공통 흐름을 프레임워크 안에 넣어두었기 때문입니다. 우리는 주로 Controller 메서드와 비즈니스 로직에 집중합니다.

DTO는 외부 요청과 내부 모델 사이의 완충지대

요청 body를 Entity에 바로 바인딩하면 외부 API 구조와 내부 도메인 구조가 강하게 묶이고, 의도하지 않은 필드가 바인딩될 위험도 커집니다. 그래서 보통 Request DTO와 Response DTO를 따로 둡니다.

record CreateMemberRequest(
    String name,
    String email
) {
}

record MemberResponse(
    Long id,
    String name
) {
}

DTO를 두면 API에 필요한 필드만 노출할 수 있고, Entity 변경이 곧바로 API 변경으로 이어지는 일을 줄일 수 있습니다.

검증은 Controller 입구에서 먼저 거르기

형식 검증은 Controller 입구에서 빠르게 처리하는 편이 좋습니다. 예를 들어 이메일 형식, 빈 값, 길이 제한 같은 규칙은 @Valid와 Bean Validation으로 표현할 수 있습니다.

record CreateMemberRequest(
    @NotBlank String name,
    @Email String email
) {
}

@PostMapping("/members")
public MemberResponse create(@Valid @RequestBody CreateMemberRequest request) {
    return memberService.create(request);
}

다만 “이미 가입된 이메일인가?”처럼 DB 조회가 필요한 규칙은 단순 형식 검증이 아니라 비즈니스 검증에 가깝습니다. 이런 검증은 Service 계층에서 다루는 편이 자연스럽습니다.

예외 처리는 한곳으로 모으기

Controller마다 try-catch를 반복하면 응답 형식이 쉽게 흩어집니다. Spring에서는 @RestControllerAdvice를 사용해 예외를 공통 응답으로 바꿀 수 있습니다.

@RestControllerAdvice
class ApiExceptionHandler {
    @ExceptionHandler(IllegalArgumentException.class)
    ResponseEntity<ErrorResponse> handleBadRequest(
        IllegalArgumentException e
    ) {
        ErrorResponse body = new ErrorResponse(e.getMessage());
        return ResponseEntity.badRequest().body(body);
    }
}

이렇게 하면 Controller는 성공 흐름에 집중하고, 실패 응답의 형식은 한곳에서 관리할 수 있습니다. body만 반환하면 별도 status 설정이 없는 한 200 응답이 될 수 있으므로 ResponseEntity나 @ResponseStatus로 HTTP status를 명시합니다.

Controller와 Service를 나누는 기준

Controller는 HTTP를 다루는 계층입니다. 요청 값을 받고, 인증 사용자 정보를 읽고, 응답 DTO를 반환합니다. Service는 HTTP와 무관한 비즈니스 흐름을 다룹니다.

간단한 기준코드에 @RequestBody, @PathVariable, HTTP status가 중요하면 웹 어댑터에 가깝고, 도메인 규칙과 트랜잭션이 중요하면 애플리케이션 서비스에 가깝습니다. 프로젝트 규모에 따라 계층 이름은 달라도 의존 방향은 분명하게 둡니다.
