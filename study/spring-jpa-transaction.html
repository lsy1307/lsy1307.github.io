트랜잭션과 JPA 영속성 컨텍스트 기본

JPA 코드는 Repository 메서드만 보면 끝나지 않습니다. 트랜잭션 안에서 영속성 컨텍스트가 어떻게 동작하는지 알아야 변경 감지와 지연 로딩을 안전하게 다룰 수 있습니다.

트랜잭션은 작업의 경계

트랜잭션은 하나의 transaction manager가 관리하는 여러 DB 작업을 한 단위로 묶습니다. 중간에 실패하면 rollback하고 성공하면 commit합니다. 서로 다른 DB나 외부 시스템까지 자동으로 원자적이 되는 것은 아닙니다. Spring에서는 보통 Service 메서드에 @Transactional을 붙여 비즈니스 작업 단위를 표현합니다.

@Service
class OrderService {
    @Transactional
    public void pay(Long orderId) {
        Order order = orderRepository.findById(orderId).orElseThrow();
        order.pay();
    }
}

Controller보다 Service에 트랜잭션을 두는 이유는 HTTP 요청 단위보다 비즈니스 작업 단위가 더 중요하기 때문입니다. 하나의 요청 안에서도 읽기만 하는 작업과 상태를 바꾸는 작업의 경계는 다를 수 있습니다.

영속성 컨텍스트: JPA가 엔티티를 관리하는 공간

영속성 컨텍스트에서 관리되는 엔티티는 같은 entity type과 ID에 대해 한 Java 객체의 동일성을 보장합니다. 트랜잭션 범위와 영속성 컨텍스트 범위는 설정에 따라 항상 같은 것은 아니므로, 정확히는 같은 영속성 컨텍스트 안이라는 조건이 중요합니다.

Member m1 = entityManager.find(Member.class, 1L);
Member m2 = entityManager.find(Member.class, 1L);

System.out.println(m1 == m2); // 같은 영속성 컨텍스트에서는 true

이 구조 덕분에 1차 캐시, 동일성 보장, 변경 감지 같은 기능이 동작합니다. 대신 트랜잭션 밖으로 엔티티를 오래 들고 나가면 지연 로딩 문제나 변경 추적 범위 문제가 생길 수 있습니다.

변경 감지: save를 매번 부르지 않아도 되는 이유

JPA provider는 영속 상태 엔티티의 변경을 추적하고 flush 시점에 필요한 update SQL을 DB로 보냅니다. flush는 commit 직전뿐 아니라 JPQL 실행 전이나 명시적 flush()에서도 일어날 수 있고, DB commit과 같은 뜻은 아닙니다. 이것을 변경 감지라고 합니다.

@Transactional
public void changeName(Long memberId, String name) {
    Member member = memberRepository.findById(memberId).orElseThrow();
    member.changeName(name);
    // 영속 상태라면 별도 save 없이 변경 감지 가능
}

그래서 JPA에서는 엔티티 메서드로 상태를 바꾸고, 트랜잭션 경계를 명확히 잡는 습관이 중요합니다.

지연 로딩은 필요한 순간에 조회한다

LAZY 연관관계는 초기화가 필요한 시점에 쿼리가 나갈 수 있습니다. 영속성 컨텍스트가 닫힌 뒤 초기화되지 않은 proxy나 collection에 접근하면 LazyInitializationException이 발생할 수 있습니다. OSIV가 켜져 있으면 Controller까지 조회가 가능할 수 있지만, 쿼리 위치와 트랜잭션 경계가 흐려지므로 이에 의존하지 않고 필요한 데이터를 조회 계층에서 명시합니다.

List<Order> orders = orderRepository.findAll();

for (Order order : orders) {
    System.out.println(order.getMember().getName());
}

N+1은 “조회 한 번 더”가 반복되는 문제

N+1은 목록 조회 1번 뒤에 각 행의 연관 데이터를 가져오는 추가 조회가 반복되는 패턴입니다. 1차 캐시와 batch fetching 때문에 정확히 N번이 아닐 수도 있지만, 데이터 수에 따라 쿼리가 늘어나는지가 핵심입니다.

  • 응답에 필요한 연관 데이터라면 fetch join이나 EntityGraph를 검토합니다.
  • 화면 전용 조회라면 DTO projection을 검토합니다.
  • 연관 객체 전체가 필요하지 않다면 ID 기반 일괄 조회나 batch fetching이 더 단순할 수 있습니다.
기준JPA 성능 문제는 “쿼리 수”와 “객체 그래프 범위”를 같이 봐야 합니다. 편한 객체 탐색이 운영 환경에서는 넓은 조회 비용으로 바뀔 수 있습니다.
