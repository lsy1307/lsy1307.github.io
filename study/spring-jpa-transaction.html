트랜잭션과 JPA 영속성 컨텍스트 기본

JPA 코드는 Repository 메서드만 보면 끝나지 않습니다. 트랜잭션 안에서 영속성 컨텍스트가 어떻게 동작하는지 알아야 변경 감지와 지연 로딩을 안전하게 다룰 수 있습니다.

트랜잭션은 작업의 경계

트랜잭션은 여러 DB 작업을 하나의 단위로 묶습니다. 중간에 실패하면 전체를 되돌리고, 성공하면 함께 반영합니다. Spring에서는 보통 Service 메서드에 @Transactional을 붙여 비즈니스 작업 단위를 표현합니다.

@Service
class OrderService {
    @Transactional
    public void pay(Long orderId) {
        Order order = orderRepository.findById(orderId).orElseThrow();
        order.pay();
    }
}

Controller보다 Service에 트랜잭션을 두는 이유는 HTTP 요청 단위보다 비즈니스 작업 단위가 더 중요하기 때문입니다. 하나의 요청 안에서도 읽기만 하는 작업과 상태를 바꾸는 작업의 경계는 다를 수 있습니다.

영속성 컨텍스트: JPA가 엔티티를 관리하는 공간

트랜잭션 안에서 조회한 엔티티는 영속성 컨텍스트에 올라갑니다. 같은 트랜잭션 안에서 같은 ID의 엔티티를 다시 조회하면, JPA는 이미 관리 중인 객체를 반환할 수 있습니다.

Member m1 = memberRepository.findById(1L).orElseThrow();
Member m2 = memberRepository.findById(1L).orElseThrow();

System.out.println(m1 == m2); // 같은 영속성 컨텍스트 안에서는 true일 수 있음

이 구조 덕분에 1차 캐시, 동일성 보장, 변경 감지 같은 기능이 동작합니다. 대신 트랜잭션 밖으로 엔티티를 오래 들고 나가면 지연 로딩 문제나 변경 추적 범위 문제가 생길 수 있습니다.

변경 감지: save를 매번 부르지 않아도 되는 이유

영속 상태의 엔티티는 트랜잭션이 끝날 때 처음 상태와 현재 상태를 비교합니다. 바뀐 값이 있으면 update SQL을 만들어 DB에 반영합니다. 이것을 변경 감지라고 합니다.

@Transactional
public void changeName(Long memberId, String name) {
    Member member = memberRepository.findById(memberId).orElseThrow();
    member.changeName(name);
    // 영속 상태라면 별도 save 없이 변경 감지 가능
}

그래서 JPA에서는 엔티티 메서드로 상태를 바꾸고, 트랜잭션 경계를 명확히 잡는 습관이 중요합니다.

지연 로딩은 필요한 순간에 조회한다

LAZY 연관관계는 실제로 사용할 때 쿼리가 나갑니다. 성능상 유리할 때도 있지만, 트랜잭션이 끝난 뒤 접근하면 예외가 날 수 있습니다. 또한 반복문 안에서 연관 객체를 계속 접근하면 예상보다 많은 쿼리가 발생할 수 있습니다.

List<Order> orders = orderRepository.findAll();

for (Order order : orders) {
    System.out.println(order.getMember().getName());
}

N+1은 “조회 한 번 더”가 반복되는 문제

N+1은 목록을 조회하는 쿼리 1번 이후, 각 행의 연관 데이터를 가져오기 위해 N번의 추가 쿼리가 발생하는 상황입니다. 데이터가 적을 때는 잘 안 보이지만, 트래픽이나 데이터가 늘면 응답 시간에 크게 영향을 줍니다.

  • 응답에 필요한 연관 데이터라면 fetch join이나 EntityGraph를 검토합니다.
  • 화면 전용 조회라면 DTO projection을 검토합니다.
  • 연관 객체 전체가 필요하지 않다면 ID 기반 조회나 일괄 조회가 더 단순할 수 있습니다.
기준JPA 성능 문제는 “쿼리 수”와 “객체 그래프 범위”를 같이 봐야 합니다. 편한 객체 탐색이 운영 환경에서는 넓은 조회 비용으로 바뀔 수 있습니다.
