커밋 단위 이미지 배포와 롤백 구조 설계

DEVLINK의 GitHub Actions–ECR–Elastic Beanstalk 배포 경험을 바탕으로, 커밋과 실행 이미지를 연결하고 같은 artifact를 재배포할 수 있는 구조를 설계했습니다.

현재 구현: 코드에서 Elastic Beanstalk까지

DEVLINK는 dev 브랜치 변경을 GitHub Actions가 감지해 Gradle 빌드, Docker 이미지 생성, ECR push, Elastic Beanstalk 배포까지 수행하도록 구성했습니다. 로컬 환경의 CPU 아키텍처와 배포 환경의 차이를 피하기 위해 이미지는 linux/amd64로 빌드했습니다.

push dev
  → Gradle build
  → docker buildx --platform linux/amd64
  → ECR push
  → Dockerrun.aws.json 패키징
  → Elastic Beanstalk deploy

수동 배포 명령을 한 흐름으로 묶었다는 점에서는 필요한 자동화였습니다. 하지만 현재 workflow는 장기 AWS access key를 사용하고, Gradle 단계에서 test를 제외하며, 완전한 commit SHA로 고정되지 않은 action과 외부 Elastic Beanstalk 배포 action을 사용합니다. 배포가 자동이라는 사실과 같은 artifact를 안전하게 다시 배포할 수 있다는 사실은 달랐습니다.

시간 기반 배포 버전과 가변 이미지

Elastic Beanstalk의 애플리케이션 버전에는 배포 시각을 넣었지만, ECR 이미지는 항상 latest 태그로 push했습니다. Dockerrun.aws.json 역시 고정된 latest를 참조했습니다.

EB version: earth-docker-20241108_21-30-00
Image:      devlink-server:latest
Update:     true

EB 버전 라벨은 배포 패키지를 식별하지만, 그 패키지가 실행할 이미지의 내용까지 고정하지 않습니다. 다음 커밋이 같은 태그를 덮어쓰면 이전 EB 버전과 새 EB 버전이 모두 동일한 문자열 latest를 가리킵니다.

식별자 불일치배포 버전은 불변처럼 보이지만 내부의 이미지 참조는 가변이었습니다. 배포 기록만으로 어떤 코드가 실행됐는지 증명할 수 없는 구조였습니다.

이전 버전 선택이 정확한 코드 복구를 보장하지 않는 이유

장애가 발생해 Elastic Beanstalk에서 과거 애플리케이션 버전을 다시 선택해도 해당 패키지에는 여전히 latest가 적혀 있습니다. 이미지 업데이트 옵션까지 켜져 있으므로 과거 패키지가 현재의 최신 이미지를 다시 받을 수 있습니다.

롤백의 단위는 “이전에 실행했던 코드와 설정의 조합”이어야 합니다. 배포 당시 이미지 digest를 알 수 없으면 다음 질문에 답하기 어렵습니다.

  • 현재 인스턴스가 어느 커밋에서 만들어진 이미지인지
  • 동일한 EB 버전을 다시 배포했을 때 같은 이미지가 실행되는지
  • 장애 직전 이미지로 정확히 돌아갈 수 있는지
  • 취약점이나 장애가 있는 이미지를 어떤 환경이 사용 중인지

현재 트리거와 목표 흐름 구분

2026년 7월 공개 저장소의 현재 배포 workflow는 dev 브랜치 push를 기준으로 동작합니다. 과거 PR #74~#76에는 pull request 트리거의 추가와 변경·되돌림 이력이 섞여 있으므로, 그 중간 상태를 현재 구현으로 설명해서는 안 됩니다.

아래는 현재 구현이 아니라 목표 흐름입니다. PR에서는 test와 build 검증만 수행하고 registry push와 배포는 막으며, 병합된 dev commit만 하나의 image를 발행하도록 분리해야 합니다. 현재 workflow의 -x test를 유지한 채 “PR에서 검증한다”고 주장할 수는 없습니다.

pull_request
  → test and build verification
  → build verification
  → no registry push
  → no deployment

push to dev
  → required checks passed
  → build once
  → immutable image push
  → deploy exact image

미적용 목표: commit에서 digest까지 하나의 식별자

Git commit SHA tag는 추적에는 유용하지만 ECR에서 같은 tag를 다시 push할 수 있으면 기술적으로 불변하지 않습니다. 목표 구조에서는 ECR tag immutability를 활성화하고, 배포 패키지는 tag보다 강한 repository@sha256:… digest를 직접 참조해야 합니다. latest를 남기더라도 사람이 현재 후보를 찾는 보조 표시에만 사용합니다.

IMAGE_TAG=${GITHUB_SHA}
TAGGED_IMAGE=$ECR_REPOSITORY_URI:$IMAGE_TAG

docker buildx build \
  --platform linux/amd64 \
  --tag "$TAGGED_IMAGE" \
  --push .

IMAGE_DIGEST=$(aws ecr describe-images ...)
DEPLOY_IMAGE="$ECR_REPOSITORY_URI@$IMAGE_DIGEST"

# 정적 JSON 안의 $DEPLOY_IMAGE는 자동 치환되지 않는다.
jq --arg image "$DEPLOY_IMAGE" \
  '.Image.Name = $image' \
  Dockerrun.template.json > Dockerrun.aws.json

정적 Dockerrun.aws.json에 "Name": "$IMAGE_URI"를 적는 것만으로 shell 변수가 치환되지는 않습니다. 위 예시처럼 jq나 envsubst로 실제 배포 파일을 생성하고, 생성된 파일의 image reference가 예상 digest와 일치하는지 검증해야 합니다.

Elastic Beanstalk 버전 라벨과 배포 manifest에는 Git commit, image digest, Dockerrun hash를 함께 남겨야 합니다. stage에서 검증한 image를 prod용으로 다시 빌드하지 않고 같은 digest를 승격해야 “검증한 것”과 “배포한 것”을 맞출 수 있습니다. 이 흐름은 현재 저장소에 적용되지 않은 목표입니다.

불변 식별자 외에 남은 운영 과제

  • AWS 인증 — 장기 access key 대신 GitHub OIDC와 환경별 최소 권한 role 사용
  • 공급망 — GitHub Action을 full commit SHA로 고정하고 third-party 배포 action의 권한과 입력 검토
  • build 재현성 — mutable base image를 digest로 고정하고 container를 non-root user로 실행
  • 배포 검증 — test를 생략하지 않고 배포 후 health·smoke test, 실패 시 자동 중단 조건 추가
  • 변경 제어 — environment approval과 deployment concurrency, 정확한 digest를 지정하는 rollback runbook 마련

Elastic Beanstalk action이 완료됐다는 사실만으로 애플리케이션이 정상이라는 뜻은 아닙니다. health와 핵심 API 확인이 통과되어야 배포 성공으로 판정하고, 실패 시 직전의 검증된 digest를 명시적으로 다시 배포해야 합니다.

자동화보다 재현 가능한 배포

CURRENTlatest 참조
TARGET IMAGEDigest 고정
TARGET PROMOTION같은 digest 이동
TARGET ROLLBACK이전 digest 지정

DEVLINK의 배포 자동화는 반복 작업을 줄였지만, 현재의 latest 참조는 코드와 실행 image 사이의 정확한 추적과 재배포를 보장하지 못합니다. 이 글에서 제안한 commit SHA tag, ECR tag immutability, digest pinning과 검증 단계는 아직 적용하지 않은 개선안입니다.

따라서 현재 성과와 목표를 분리해 기록합니다. 현재는 dev push에서 ECR·Elastic Beanstalk까지 자동화되어 있고, 다음 단계는 “무엇이 올라갔는지 식별할 수 있는가”와 “같은 artifact를 검증·승격·롤백할 수 있는가”를 구현으로 증명하는 일입니다.
