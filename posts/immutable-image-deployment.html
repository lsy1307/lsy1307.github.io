Deployment · Reproducibility

latest로 배포하면 롤백할 수 없는 이유

DEVLINK의 GitHub Actions–ECR–Elastic Beanstalk 파이프라인을 다시 검토하며, 배포 버전과 실행 이미지가 일치해야 하는 이유를 정리했습니다.

코드에서 Elastic Beanstalk까지

DEVLINK는 dev 브랜치 변경을 GitHub Actions가 감지해 Gradle 빌드, Docker 이미지 생성, ECR push, Elastic Beanstalk 배포까지 수행하도록 구성했습니다. 로컬 환경의 CPU 아키텍처와 배포 환경의 차이를 피하기 위해 이미지는 linux/amd64로 빌드했습니다.

push dev
  → Gradle build
  → docker buildx --platform linux/amd64
  → ECR push
  → Dockerrun.aws.json 패키징
  → Elastic Beanstalk deploy

수동 배포 명령을 한 흐름으로 묶었다는 점에서는 필요한 자동화였습니다. 하지만 배포가 자동이라는 사실과 같은 코드를 다시 배포할 수 있다는 사실은 달랐습니다.

시간 기반 배포 버전과 가변 이미지

Elastic Beanstalk의 애플리케이션 버전에는 배포 시각을 넣었지만, ECR 이미지는 항상 latest 태그로 push했습니다. Dockerrun.aws.json 역시 고정된 latest를 참조했습니다.

EB version: earth-docker-20241108_21-30-00
Image:      devlink-server:latest
Update:     true

EB 버전 라벨은 배포 패키지를 식별하지만, 그 패키지가 실행할 이미지의 내용까지 고정하지 않습니다. 다음 커밋이 같은 태그를 덮어쓰면 이전 EB 버전과 새 EB 버전이 모두 동일한 문자열 latest를 가리킵니다.

식별자 불일치배포 버전은 불변처럼 보이지만 내부의 이미지 참조는 가변이었습니다. 배포 기록만으로 어떤 코드가 실행됐는지 증명할 수 없는 구조였습니다.

이전 버전 선택이 이전 코드 복구가 아닌 이유

장애가 발생해 Elastic Beanstalk에서 과거 애플리케이션 버전을 다시 선택해도 해당 패키지에는 여전히 latest가 적혀 있습니다. 이미지 업데이트 옵션까지 켜져 있으므로 과거 패키지가 현재의 최신 이미지를 다시 받을 수 있습니다.

롤백의 단위는 “이전에 실행했던 코드와 설정의 조합”이어야 합니다. 배포 당시 이미지 digest를 알 수 없으면 다음 질문에 답하기 어렵습니다.

  • 현재 인스턴스가 어느 커밋에서 만들어진 이미지인지
  • 동일한 EB 버전을 다시 배포했을 때 같은 이미지가 실행되는지
  • 장애 직전 이미지로 정확히 돌아갈 수 있는지
  • 취약점이나 장애가 있는 이미지를 어떤 환경이 사용 중인지

PR 검증과 배포의 분리

초기 워크플로는 dev 브랜치 push뿐 아니라 해당 브랜치를 대상으로 한 pull request에서도 전체 배포 job을 실행하도록 구성했습니다. 이 경우 아직 병합되지 않은 커밋이 latest를 덮어쓰고 개발 환경에 배포될 수 있습니다.

PR에서는 코드가 배포 가능한지 검증하고, 병합된 커밋만 이미지를 발행해야 합니다. 두 이벤트가 같은 job을 공유하면 검증 대상과 실제 배포 대상의 경계가 흐려집니다.

pull_request
  → test
  → build verification
  → no registry push
  → no deployment

push to dev
  → test
  → build once
  → immutable image push
  → deploy exact image

커밋에서 실행 이미지까지 하나의 식별자

다시 구성한다면 이미지 태그에 Git commit SHA를 사용하고, 배포 패키지가 해당 태그나 digest를 직접 참조하도록 만듭니다. latest는 사람이 현재 버전을 찾기 위한 보조 태그로는 쓸 수 있지만 배포의 기준으로 사용하지 않습니다.

IMAGE_TAG=${GITHUB_SHA}
IMAGE_URI=$ECR_REGISTRY/devlink-server:$IMAGE_TAG

docker buildx build \
  --platform linux/amd64 \
  --tag "$IMAGE_URI" \
  --push .

# Dockerrun.aws.json
"Name": "$IMAGE_URI"

Elastic Beanstalk 버전 라벨에도 짧은 commit SHA를 포함하면 Git 커밋, ECR 이미지, EB 애플리케이션 버전을 한 방향으로 추적할 수 있습니다. 더 강하게 고정하려면 ECR이 반환한 image digest를 배포 메타데이터로 보관합니다.

stage에서 검증한 이미지를 prod용으로 다시 빌드하지 않는 것도 중요합니다. 같은 digest를 환경별로 승격해야 “검증한 것”과 “배포한 것”이 같아집니다. 빌드 시각이 다른 이미지는 소스가 같아도 의존성이나 빌드 환경 차이로 내용이 달라질 수 있습니다.

자동화보다 재현 가능한 배포

CIPR은 검증만
IMAGECommit SHA 고정
PROMOTION같은 digest 이동
ROLLBACK이전 이미지 지정

DEVLINK의 배포 자동화는 반복 작업을 줄였지만, latest는 코드와 실행 환경 사이의 추적성을 끊었습니다. 파이프라인의 완성도는 단계 수가 아니라 특정 커밋을 동일한 이미지로 다시 실행하고 되돌릴 수 있는지로 판단해야 했습니다.

이후 배포 설계를 볼 때는 “자동으로 올라가는가”보다 “무엇이 올라갔는지 식별할 수 있는가”를 먼저 확인하게 됐습니다.
