Load Test · Infrastructure

운영과 닮은 부하 테스트 환경을 필요할 때만 만들기

운영 데이터 규모를 반영한 RDS와 전용 k6 실행기를 일회성으로 만들고, 실행이 끝나면 제거하는 자동화 흐름을 설계했습니다.

성능 테스트의 결과는 환경만큼만 믿을 수 있다

작은 로컬 DB와 stage 서버에서 얻은 응답 시간은 운영 환경의 인덱스 선택도, 데이터 분포, 네트워크 경로를 충분히 반영하지 못합니다. 반대로 운영 DB를 직접 대상으로 테스트하는 것은 데이터와 가용성 모두에 위험합니다.

필요한 것은 운영과 닮았지만 운영과 격리되고, 테스트할 때만 비용이 발생하며, 누구나 같은 순서로 다시 만들 수 있는 환경이었습니다.

설계 목표운영 데이터 규모의 현실성, 네트워크 격리, 반복 가능한 실행, 사용 후 자원 정리를 하나의 생명주기로 묶습니다.

Start · Run · Stop으로 책임을 나눴다

하나의 거대한 workflow 대신 세 단계로 분리했습니다. 환경 준비와 테스트 실행, 정리는 실패 가능성과 반복 주기가 다르기 때문입니다.

  1. Start — Terraform으로 부하 테스트 RDS와 필요한 네트워크 자원 생성, 선택적으로 stage 애플리케이션을 loadtest profile로 전환
  2. Run — 전용 EC2를 만들고 SSM으로 k6 파일을 동기화한 뒤 입력값에 따라 테스트 실행
  3. Stop — stage 설정을 복구하고 RDS와 부하 발생기를 제거

세 workflow는 같은 concurrency group을 사용하고 실행 중인 작업을 자동 취소하지 않도록 했습니다. 환경 생성 도중 다른 종료 작업이 끼어드는 경쟁 상태를 막기 위한 선택입니다.

운영 RDS의 최신 자동 스냅샷에서 시작한다

부하 테스트 RDS는 운영 RDS의 최신 자동 스냅샷으로 복원합니다. 애플리케이션이 실제와 가까운 데이터 분포를 대상으로 쿼리를 실행하게 하되, 원본 운영 DB와는 완전히 분리합니다.

data "aws_db_snapshot" "latest_prod" {
  db_instance_identifier = var.prod_rds_identifier
  most_recent             = true
  snapshot_type           = "automated"
}

resource "aws_db_instance" "load_test" {
  snapshot_identifier = data.aws_db_snapshot.latest_prod.id
  publicly_accessible = false
  storage_encrypted   = true
  skip_final_snapshot = true
}

RDS는 public 접근을 허용하지 않고 prod·stage API 서버의 Security Group에서 오는 MySQL 트래픽만 받습니다. 테스트용이라고 해서 네트워크 경계를 느슨하게 만들지 않았습니다.

stage 서버와 부하 발생기를 분리했다

k6를 stage 애플리케이션 서버에서 실행하면 애플리케이션과 부하 도구가 CPU·메모리·네트워크를 경쟁하게 됩니다. 측정 대상이 부하 생성 비용까지 떠안는 셈입니다.

별도의 EC2를 부하 발생기로 만들고, 테스트가 끝나면 기본적으로 제거합니다. 인스턴스는 암호화된 gp3 볼륨을 사용하고 IMDSv2를 강제했습니다. 직접 SSH 접속 대신 SSM RunCommand로 파일 동기화와 실행을 제어합니다.

  • 테스트 대상과 부하 생성 자원의 경합 제거
  • SSH key 전달 없이 IAM과 SSM으로 실행
  • xk6 및 remote-write extension 버전 고정
  • Prometheus URL이 있을 때만 remote-write 활성화

workflow 입력을 셸 코드로 만들지 않는다

VUs, iteration 수, 최대 실행 시간과 대상 URL은 workflow_dispatch 입력으로 받습니다. 입력 표현식을 셸 명령 안에 직접 삽입하지 않고 환경 변수로 전달한 뒤 Bash 배열에 인자로 추가했습니다.

env:
  VUS: ${{ inputs.vus }}
  MAX_DURATION: ${{ inputs.max_duration }}

run: |
  args=(
    --vus "$VUS"
    --max-duration "$MAX_DURATION"
  )
  bash scripts/load_test/run_k6.sh "${args[@]}"

원격 명령 JSON도 jq --arg와 shell escaping을 이용해 구성했습니다. 운영 자동화에서 편리한 문자열 보간이 곧 명령 주입 경로가 될 수 있기 때문입니다.

실패해도 정리되는 실행 흐름

실행 스크립트는 trap으로 종료 시 부하 발생기 제거를 시도합니다. k6가 실패해도 원래 종료 코드를 보존하고 cleanup을 수행하며, 필요할 때만 실행기를 남길 수 있는 옵션을 제공합니다.

SSM 명령은 성공할 때까지 무한히 기다리지 않습니다. 상태를 주기적으로 확인하고 제한 시간을 넘으면 원격 출력을 남긴 뒤 실패 처리합니다. 자동화의 완료 조건과 실패 조건을 모두 코드에 넣은 셈입니다.

이번 작업의 핵심은 k6를 실행하는 명령이 아니라 신뢰할 수 있는 측정 환경의 생명주기를 만드는 일이었습니다. 성능 테스트를 반복 가능한 운영 작업으로 만들자 결과를 해석할 때의 불확실성도 줄어들었습니다.
