MySQL과 MongoDB 사이의 트랜잭션 경계

DEVLINK의 공간 데이터에 MongoDB를 적용한 구현을 다시 살펴보며, 저장소를 나누는 것보다 어려웠던 소유권과 실패 복구의 경계를 정리했습니다.

관계 데이터와 공간 데이터의 분리

DEVLINK는 사용자·프로젝트·팀처럼 관계와 무결성이 중요한 데이터를 MySQL에 저장하고, 협업 공간의 채널·타일·오브젝트·메모·PDF는 MongoDB에 저장했습니다. 공간 오브젝트는 종류마다 속성이 다르고 위치와 회전값이 자주 바뀌기 때문에 문서 모델이 변경을 수용하기 쉽다고 판단했습니다.

저장 기술을 두 개 사용했다는 사실보다 중요한 것은 각 저장소가 무엇을 책임지는지였습니다. MySQL은 사용자와 팀의 관계를, MongoDB는 채널 안에 배치되는 콘텐츠를 담당했습니다.

초기 기준스키마가 유연하다는 이유만으로 MongoDB를 선택한 것이 아니라, 독립적으로 수정되는 공간 객체의 접근 패턴을 저장소 경계로 삼았습니다.

채널 ID로 연결한 독립 컬렉션

타일·오브젝트·메모·PDF를 하나의 거대한 채널 문서에 포함하지 않고 각각의 컬렉션으로 분리했습니다. 각 문서는 channelId를 인덱스로 가지며, 오브젝트와 메모는 위치값을 내장 문서로 저장합니다.

@Document(collection = "memoInfo")
class MemoInfo {
    @Id
    String memoId;

    @Indexed
    String channelId;

    String memoText;
    Position position;
}

이 구조에서는 메모 한 개나 오브젝트 한 개를 독립적으로 수정하기 쉽고, 객체 종류가 늘어날 때 채널 문서 전체의 형태를 바꾸지 않아도 됩니다. 반면 채널 하나를 완전하게 읽거나 삭제하려면 여러 컬렉션을 순회해야 합니다. 쓰기 독립성을 얻는 대신 집계와 수명주기 관리 비용을 애플리케이션이 부담한 셈입니다.

하나의 애너테이션, 두 개의 원자성

채널 생성 흐름은 MongoDB에 채널을 저장한 다음 MySQL의 사용자 엔티티에 해당 channelId를 추가했습니다. 코드에서는 하나의 서비스 메서드에 @Transactional을 붙였지만, 이것만으로 JPA 트랜잭션과 MongoDB 트랜잭션이 하나가 되지는 않습니다.

@Transactional
public ChannelResponse registerChannel(Request request, String accountId) {
    Channel channel = channelRepository.save(
        new Channel(request.getChannelName())
    );

    User user = userService.findByAccountId(accountId);
    user.getChannelList().add(channel.getChannelId());
    return new ChannelResponse(channel);
}

현재 구성에는 두 저장소를 함께 조정하는 분산 트랜잭션이 없습니다. MongoDB 저장 후 사용자 조회나 MySQL 커밋이 실패하면 채널만 남을 수 있습니다. 반대 순서로 구현해도 문제는 사라지지 않고, 이번에는 존재하지 않는 채널 ID가 사용자에게 남을 수 있습니다.

트랜잭션 애너테이션이 있다는 사실과 비즈니스 작업 전체가 원자적이라는 사실을 구분해야 했습니다. 저장소가 둘이 되는 순간 실패 지점도 저장소 사이에 하나 더 생깁니다.

애플리케이션 cascade의 실패 구간

채널 삭제 역시 타일, 오브젝트, 메모, PDF를 차례로 지운 뒤 마지막에 채널을 삭제하는 방식이었습니다.

deleteTiles(channelId);
deleteObjects(channelId);
deleteMemos(channelId);
deletePdfs(channelId);
deleteChannel(channelId);

RDB의 외래 키와 cascade가 아니라 서비스 코드가 참조 무결성을 관리합니다. 세 번째 단계에서 장애가 나면 앞의 두 컬렉션은 삭제됐지만 뒤의 데이터는 남습니다. 재시도할 때 이미 삭제된 데이터가 있어도 안전해야 하며, 누락된 문서를 정상 상태로 볼 것인지 복구 대상으로 볼 것인지도 정해야 합니다.

  • 각 삭제 연산을 존재 여부와 무관하게 성공하는 멱등 연산으로 구성
  • 삭제 요청에 작업 ID를 부여해 중복 실행과 진행 상태 추적
  • 채널을 즉시 지우지 않고 DELETING 상태로 전환
  • 부분 실패를 재처리하고 고아 문서를 탐지하는 정합성 작업 운영

저장소보다 먼저 정하는 원본

다시 설계한다면 먼저 채널 소유 관계의 source of truth를 한 곳으로 정합니다. 관계가 MySQL의 책임이라면 MySQL에서 사용자–채널 관계와 Outbox 이벤트를 같은 트랜잭션으로 기록하고, 소비자가 MongoDB의 공간을 멱등하게 생성하도록 만들 수 있습니다.

MySQL transaction
  ├─ user_channel 생성
  └─ outbox(CHANNEL_CREATE_REQUESTED) 기록

commit
  └─ consumer가 MongoDB channel 생성
       └─ 성공/실패 상태 갱신

반대로 MongoDB를 채널의 원본으로 정한다면 Channel 문서에 ownerUserId를 저장하고, MySQL 사용자 엔티티의 문자열 ID 목록을 제거하는 편이 낫습니다. 양쪽에 같은 관계를 저장하고 동시에 맞추려는 구조보다 한쪽을 원본, 다른 쪽을 projection으로 만드는 구조가 실패를 설명하기 쉽습니다.

모든 공간 객체를 채널 문서에 포함하는 방법도 있습니다. 단일 문서 갱신의 원자성을 얻지만, 문서 크기 증가와 동시 수정 충돌이 커집니다. 객체별 수정 빈도가 높았던 DEVLINK에서는 무조건적인 embedding보다 독립 컬렉션과 명시적인 삭제 워크플로가 더 현실적인 선택입니다.

폴리글랏 퍼시스턴스의 판단 기준

DATA MODEL접근 패턴 기준
OWNERSHIP원본은 한 곳
FAILURE부분 성공 명시
RECOVERY멱등성과 재처리

DEVLINK에서 MongoDB를 사용한 경험의 핵심은 문서형 DB의 CRUD가 아니었습니다. 저장소를 나누면 유연성만 늘어나는 것이 아니라 트랜잭션 밖의 상태와 복구 책임도 함께 늘어난다는 점이었습니다.

이후 여러 저장소를 연결할 때는 기술의 장점보다 먼저 원본 데이터의 위치, 중간 실패 후의 상태, 같은 요청을 다시 실행해도 되는지를 확인하게 됐습니다.
