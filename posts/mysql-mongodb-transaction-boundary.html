MySQL과 MongoDB 사이의 트랜잭션 경계

DEVLINK에서 실제로 구현한 MySQL·MongoDB 분리와 순차 삭제를 먼저 설명하고, Outbox·상태 전이·재처리는 당시 적용하지 않은 재설계안으로 분리해 정리했습니다.

당시 구현: 관계 데이터와 공간 데이터의 분리

DEVLINK는 사용자·프로젝트·팀처럼 관계와 무결성이 중요한 데이터를 MySQL에 저장하고, 협업 공간의 채널·타일·오브젝트·메모·PDF는 MongoDB에 저장했습니다. 공간 오브젝트는 종류마다 속성이 다르고 위치와 회전값이 자주 바뀌기 때문에 문서 모델이 변경을 수용하기 쉽다고 판단했습니다.

저장 기술을 두 개 사용했다는 사실보다 중요한 것은 각 저장소가 무엇을 책임지는지였습니다. MySQL은 사용자와 팀의 관계를, MongoDB는 채널 안에 배치되는 콘텐츠를 담당했습니다.

구현 범위독립 컬렉션과 channelId 연결, 두 저장소를 차례로 갱신하는 서비스 코드는 당시 구현입니다. 아래의 Outbox, 작업 상태, 재시도·정합성 배치는 구현 성과가 아니라 현재 시점의 재설계 제안입니다.

채널 ID로 연결한 독립 컬렉션

타일·오브젝트·메모·PDF를 하나의 거대한 채널 문서에 포함하지 않고 각각의 컬렉션으로 분리했습니다. 각 문서는 channelId를 인덱스로 가지며, 오브젝트와 메모는 위치값을 내장 문서로 저장합니다.

@Document(collection = "memoInfo")
class MemoInfo {
    @Id
    String memoId;

    @Indexed
    String channelId;

        String memoText;
    Position position;
}

운영 주의: @Indexed는 인덱스 선언 의도를 나타낼 뿐이며, 자동 인덱스 생성 설정이나 배포 과정의 명시적 프로비저닝이 실행돼야 실제 MongoDB 인덱스가 만들어지므로 운영 환경에서는 인덱스 목록과 실행 계획으로 적용 여부를 확인해야 합니다.

이 구조에서는 메모 한 개나 오브젝트 한 개를 독립적으로 수정하기 쉽고, 객체 종류가 늘어날 때 채널 문서 전체의 형태를 바꾸지 않아도 됩니다. 반면 채널 하나를 완전하게 읽거나 삭제하려면 여러 컬렉션을 순회해야 합니다. 쓰기 독립성을 얻는 대신 집계와 수명주기 관리 비용을 애플리케이션이 부담한 셈입니다.

당시 구현: 하나의 애너테이션, 분리된 원자성

채널 생성 흐름은 MongoDB에 채널을 저장한 다음 MySQL의 사용자 엔티티에 해당 channelId를 추가했습니다. 코드에서는 하나의 서비스 메서드에 @Transactional을 붙였지만, 이것만으로 JPA 트랜잭션과 MongoDB 트랜잭션이 하나가 되지는 않습니다.

@Transactional
public ChannelResponse registerChannel(Request request, String accountId) {
    Channel channel = channelRepository.save(
        new Channel(request.getChannelName())
    );

    User user = userService.findByAccountId(accountId);
    user.getChannelList().add(channel.getChannelId());
    return new ChannelResponse(channel);
}

@Transactional은 설정에서 선택된 트랜잭션 매니저의 범위에 적용될 뿐, JPA와 MongoDB 작업을 자동으로 하나의 원자적 작업으로 합치지 않습니다. 당시 구성에는 두 저장소를 함께 조정하는 분산 트랜잭션이나 보상 워크플로가 없었습니다. MongoDB 저장 후 사용자 조회나 MySQL 커밋이 실패하면 채널만 남을 수 있습니다. 반대 순서로 구현해도 문제는 사라지지 않고, 이번에는 존재하지 않는 채널 ID가 사용자에게 남을 수 있습니다.

트랜잭션 애너테이션이 있다는 사실과 비즈니스 작업 전체가 원자적이라는 사실을 구분해야 했습니다. 저장소가 둘이 되는 순간 실패 지점도 저장소 사이에 하나 더 생깁니다.

당시 구현: 애플리케이션 cascade의 실패 구간

채널 삭제 역시 타일, 오브젝트, 메모, PDF를 차례로 지운 뒤 마지막에 채널을 삭제하는 방식이었습니다.

deleteTiles(channelId);
deleteObjects(channelId);
deleteMemos(channelId);
deletePdfs(channelId);
deleteChannel(channelId);

RDB의 외래 키와 cascade가 아니라 서비스 코드가 참조 무결성을 관리합니다. 세 번째 단계에서 장애가 나면 앞의 두 컬렉션은 삭제됐지만 뒤의 데이터는 남을 수 있습니다. 당시 구현에는 이 부분 성공을 추적하는 작업 상태나 자동 재처리 흐름이 없었습니다.

아래 항목은 재설계안다음 네 항목은 당시 적용한 기능이 아니라, 순차 삭제의 실패를 검토한 뒤 도출한 보강 방향입니다.
  • 각 삭제 연산을 존재 여부와 무관하게 성공하는 멱등 연산으로 구성
  • 삭제 요청에 작업 ID를 부여해 중복 실행과 진행 상태 추적
  • 채널을 즉시 지우지 않고 DELETING 상태로 전환
  • 부분 실패를 재처리하고 고아 문서를 탐지하는 정합성 작업 운영

재설계안: 저장소보다 먼저 정하는 원본

이 절의 Outbox와 projection 구조는 DEVLINK에 적용한 구현이 아닙니다. 다시 설계한다면 먼저 채널 소유 관계의 source of truth를 한 곳으로 정하겠다는 제안입니다. 관계가 MySQL의 책임이라면 MySQL에서 사용자–채널 관계와 Outbox 이벤트를 같은 트랜잭션으로 기록하고, 소비자가 MongoDB의 공간을 멱등하게 생성하도록 설계할 수 있습니다.

// IMPROVEMENT DESIGN — 재처리 가능한 상태 전이
MySQL transaction
  ├─ user_channel 생성
  └─ outbox(CHANNEL_CREATE_REQUESTED) 기록

commit
  └─ consumer가 MongoDB channel 생성
       └─ 성공/실패 상태 갱신

반대로 MongoDB를 채널의 원본으로 정한다면 Channel 문서에 ownerUserId를 저장하고, MySQL 사용자 엔티티의 문자열 ID 목록을 제거하는 방안도 검토할 수 있습니다. 양쪽에 같은 관계를 저장하고 동시에 맞추려는 구조보다 한쪽을 원본, 다른 쪽을 projection으로 만드는 편이 실패 상태를 설명하기 쉽습니다.

모든 공간 객체를 채널 문서에 포함하는 방법도 후보입니다. 단일 문서 범위의 원자성을 활용할 수 있지만, 문서 크기와 동시 수정 충돌을 함께 검토해야 합니다. 당시의 독립 컬렉션을 유지한다면 명시적인 삭제 상태와 재처리 흐름이 추가로 필요하다는 것이 현재의 판단입니다.

폴리글랏 퍼시스턴스의 판단 기준

IMPLEMENTED저장소·컬렉션 분리
KNOWN GAP부분 성공 추적 없음
REDESIGN원본과 상태 명시
NOT IMPLEMENTEDOutbox·재처리

DEVLINK에서 실제로 구현한 것은 MySQL·MongoDB의 역할 분리와 애플리케이션의 순차 처리입니다. 그 구현은 다중 저장소 전체의 원자성이나 자동 복구를 제공하지 않았습니다. Outbox, DELETING 상태, 멱등 소비자와 정합성 작업은 이 한계를 바탕으로 정리한 재설계안입니다.

이후 여러 저장소를 연결할 때는 기술의 장점보다 먼저 원본 데이터의 위치, 중간 실패 후의 상태, 같은 요청을 다시 실행해도 되는지를 확인하게 됐습니다. 회고에서 설계안을 제시할 때도 당시 구현과 적용하지 않은 대안을 분리해 설명해야 합니다.
