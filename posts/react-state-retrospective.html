React 서버 상태 동기화와 검색 상태 설계 회고

댓글 변경 후 두 API를 재검증한 선택과 멤버 검색의 원본·결과 상태 분리를 돌아보며, 당시 구현의 장점과 debounce의 한계를 함께 정리했습니다.

한 화면을 구성하는 두 서버 상태

공지 상세 화면은 공지 정보와 댓글 목록을 서로 다른 API에서 가져왔습니다. 댓글을 추가하거나 삭제하면 댓글 배열뿐 아니라 공지 정보가 가진 댓글 수도 함께 바뀝니다.

하나의 사용자 행동이 두 서버 상태에 영향을 주지만, 프론트엔드에서는 각각 별도 state로 관리했습니다. 댓글 목록만 수정하면 상세 영역의 댓글 수가 이전 값으로 남아 같은 화면 안에서 서로 다른 상태가 보일 수 있었습니다.

상태 경계API가 두 개라는 사실보다 중요한 것은 두 응답이 같은 도메인 변경의 영향을 받는다는 점이었습니다.

Mutation 이후 동시 재검증

댓글 작성 요청이 성공하면 입력값을 비우고, 공지 상세와 댓글 목록을 Promise.all로 함께 다시 조회했습니다. 삭제 후에도 같은 재검증 함수를 호출했습니다.

await createComment(postId, comment);
setComment("");

await Promise.all([
  fetchNoticeDetail(),
  fetchNoticeComments()
]);

서버 응답을 source of truth로 사용하기 때문에 클라이언트가 댓글 수 계산 규칙을 복제하지 않아도 됩니다. 작성·삭제 권한이나 soft delete 정책이 서버에서 적용돼도 최종 응답 기준으로 화면을 다시 맞출 수 있습니다.

대신 mutation마다 두 번의 GET 요청이 추가되고, 조회 중 loading 처리와 실패 정책이 필요합니다. 상세 조회는 성공했지만 댓글 조회가 실패하는 부분 성공도 고려해야 합니다. 작은 서비스에서는 단순하고 안전한 선택이었지만 규모가 커지면 cache invalidation 정책으로 발전시킬 지점입니다.

원본 목록과 화면 결과의 분리

멤버 검색에서는 서버에서 받은 전체 목록을 allMembers에 보존하고, 화면에 표시할 목록을 results로 분리했습니다.

state = {
  search: "",
  allMembers: serverMembers,
  results: serverMembers
};

검색할 때는 원본 목록을 필터링해 results만 바꾸고, 검색어가 비면 allMembers를 다시 표시했습니다. 검색 해제 시 API를 재호출하지 않고 원본 목록으로 복귀할 수 있다는 장점이 있었습니다.

다만 results는 allMembers와 검색어에서 계산할 수 있는 파생 상태입니다. 두 배열을 별도 state로 저장하면 원본 목록 갱신 시 결과 목록을 함께 맞춰야 하고, 동기화 지점을 놓칠 가능성이 생깁니다.

300ms debounce의 실제 효과

입력값을 300ms 지연하는 custom debounce를 적용했지만 검색은 입력 변경마다 실행되지 않고 Enter 또는 검색 버튼 클릭 시에만 실행됐습니다. 이 구조에서는 연속 입력 중 반복되는 API 요청이나 필터 연산 자체가 없으므로 debounce가 줄여 주는 작업도 거의 없습니다.

const debouncedSearch = debounce(search, 300);

// 검색 실행은 입력 변경이 아니라 클릭·Enter에만 연결
const handleSearch = () => {
  filter(allMembers, debouncedSearch);
};

오히려 마지막 입력 후 300ms 안에 버튼을 누르면 갱신 전 검색어가 사용될 수 있습니다. debounce는 값을 늦추는 것만으로 최적화가 되지 않습니다. 지연된 값이 실제 비용이 큰 작업의 실행 조건과 연결돼야 의미가 있습니다.

핵심 구분debounce는 최적화 기법 자체가 아니라 작업 실행 시점을 조절하는 스케줄링 방식입니다.

로컬 검색과 서버 검색의 다른 설계

작은 로컬 목록

멤버 목록이 이미 메모리에 있고 크기가 작다면 debounce 없이 검색어와 원본 목록에서 결과를 계산하는 편이 단순합니다.

const filteredMembers = useMemo(() => {
  const query = search.trim().toLowerCase();
  if (!query) return allMembers;

  return allMembers.filter(member =>
    member.nickname.toLowerCase().includes(query)
  );
}, [allMembers, search]);

파생 배열을 state로 저장하지 않기 때문에 원본과 결과가 어긋날 지점도 줄어듭니다. 목록이 매우 커서 필터 비용이 실제로 문제가 된 뒤에 deferred value나 virtualization을 검토할 수 있습니다.

서버 검색

입력마다 네트워크 요청이 필요한 검색이라면 debounced value 변경을 effect의 실행 조건으로 연결해야 합니다. 이전 요청보다 나중 요청이 먼저 도착하는 race condition도 취소나 요청 식별자로 방어해야 합니다.

useEffect(() => {
  if (!debouncedQuery) return;

  const controller = new AbortController();
  searchMembers(debouncedQuery, controller.signal);

  return () => controller.abort();
}, [debouncedQuery]);

서버 상태와 파생 상태의 구분

SERVER STATEMutation 후 재검증
DERIVED STATE저장보다 계산
LOCAL SEARCH즉시 필터링
REMOTE SEARCHdebounce + 취소

당시 구현에서 Promise.all 재조회는 서버를 source of truth로 유지하려는 명확한 선택이었습니다. 반면 검색 결과를 별도 state로 저장하고 클릭 기반 검색에 debounce를 적용한 부분은 복잡성에 비해 효과가 작았습니다.

같은 React state라도 서버에서 가져온 원본인지, 다른 값으로부터 계산되는 파생 결과인지, 사용자 입력의 임시 값인지에 따라 관리 방식이 달라집니다. 구현 패턴보다 먼저 상태의 성격을 구분하는 것이 더 중요한 기준이 됐습니다.
