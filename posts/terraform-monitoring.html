Infrastructure as Code

수동 모니터링 서버를 재현 가능한 코드로 옮기기

EC2, Security Group, Nginx와 관측 도구 구성을 Terraform 모듈로 분리해 변경 범위를 명확하게 만든 경험입니다.

GRAFANA DASHBOARD / ARCHITECTURE · 16:9
CONTENTS왜 IaC인가모듈 경계네트워크운영회고

서버 안에만 남은 구성 정보

Grafana, Loki, Prometheus 기반 모니터링 서버는 수동 설정에 의존하고 있었습니다. EC2 생성과 Security Group 규칙, Nginx reverse proxy 구성이 콘솔과 서버 내부에 흩어져 재현하기 어려웠습니다.

당장 동작하는 것과 다시 만들 수 있는 것은 다릅니다. 장애나 이전이 발생했을 때 같은 구성을 복구할 수 있도록 인프라의 현재 상태를 코드로 옮기기로 했습니다.

monitoring_stack 모듈 분리

애플리케이션 인프라와 관측 인프라는 변경 주기가 다릅니다. 모니터링 구성을 monitoring_stack 모듈로 분리하고 EC2, Security Group, Nginx에 필요한 값을 입력과 출력으로 드러냈습니다.

module "monitoring_stack" {
  source = "./modules/monitoring-stack"

  vpc_id          = module.network.vpc_id
  public_subnet_id = module.network.public_subnet_id
  instance_type    = var.monitoring_instance_type
}

모듈 경계를 나누자 애플리케이션 배포와 모니터링 인프라 변경을 독립적으로 검토할 수 있었습니다. 무엇이 바뀌는지 plan 단계에서 확인할 수 있다는 점도 수동 작업과 큰 차이였습니다.

접근 경로의 명시적 정의

Security Group 규칙과 Nginx reverse proxy 설정을 코드에 포함했습니다. 외부에 직접 노출할 포트와 내부 컴포넌트 간 통신을 구분하고, 접근 지점을 Nginx로 모았습니다.

설계 기준도구를 설치하는 것보다 “누가 어떤 경로로 접근할 수 있는가”를 코드 리뷰에서 확인할 수 있게 하는 데 초점을 맞췄습니다.

plan 기반 변경 검토

이후 인프라 변경은 코드 수정, terraform plan 검토, 적용의 순서로 진행할 수 있게 됐습니다. 수동 설정의 숨은 차이를 줄이고 Git 변경 이력으로 의사결정을 추적할 수 있습니다.

  • 모니터링 환경을 같은 코드로 다시 생성 가능
  • 리소스 변경 범위를 적용 전에 확인
  • 관측 인프라와 애플리케이션 인프라의 변경 책임 분리

생성 자동화보다 변경 관리

Terraform을 적용한 가장 큰 이유는 서버 생성 시간을 몇 분 줄이기 위해서가 아니었습니다. 현재 상태와 원하는 상태의 차이를 읽고, 팀이 같은 변경 내용을 검토하게 만드는 것이 핵심이었습니다. 재현 가능성은 그 과정에서 얻은 가장 실질적인 운영 안정성이었습니다.
