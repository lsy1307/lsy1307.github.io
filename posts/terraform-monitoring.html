수동 AWS 인프라에서 Terraform으로

prod·stage·monitoring·load_test·global을 독립 root와 S3 state로 나눈 구현 기록입니다. 현재 모니터링 영역은 EC2·Security Group·Docker 설치와 수동 Nginx 설정 기반까지 코드화됐으며, Grafana·Loki·Prometheus 구성 전체가 Terraform으로 관리되는 상태는 아닙니다.

동작하지만 재현할 수 없는 인프라

초기 AWS 환경은 콘솔에서 EC2와 Security Group을 만들고, 서버에 접속해 Docker와 Nginx를 설정하는 방식이었습니다. 서비스는 동작했지만 현재 구성이 왜 그렇게 되었는지, 다른 환경에 같은 구성을 만들려면 무엇이 필요한지 한눈에 알기 어려웠습니다.

운영 이슈가 생길 때마다 콘솔의 현재 값과 서버 내부 파일, 팀원의 기억을 함께 확인해야 했습니다. stage와 prod가 비슷해 보여도 작은 설정 차이가 누적됐고, 변경 전후를 PR에서 검토할 수도 없었습니다.

도입 목적리소스 생성 자동화보다 재현 가능한 현재 상태, 변경 전 diff, 환경 간 차이의 명시적 관리가 먼저였습니다.

명령 모음보다 선언적 상태

AWS CLI 스크립트도 반복 작업을 줄일 수 있지만 이미 존재하는 리소스와 원하는 상태의 차이를 지속적으로 관리하기 어렵습니다. Terraform은 리소스 의존성을 그래프로 계산하고, plan에서 변경 범위를 먼저 보여준다는 점이 요구사항과 맞았습니다.

  • 현재 코드와 실제 리소스의 차이를 plan에서 확인
  • EC2·RDS·Security Group·S3·Lambda의 의존 순서 계산
  • Git 변경 이력과 PR을 통한 인프라 의사결정 추적
  • 공통 모듈과 환경별 입력값을 통한 중복 감소

목표는 모든 운영 명령을 Terraform 안에 넣는 것이 아니라, 수명주기와 의존성을 관리해야 하는 리소스를 선언적 상태로 옮기는 것이었습니다.

환경별 root module과 state 분리

prod, stage, monitoring, load_test, global을 각각 독립된 root module로 나눴습니다. 모든 환경을 하나의 거대한 state에 넣으면 작은 변경도 전체 인프라의 blast radius를 넓히기 때문입니다.

environment/
├── prod/
├── stage/
├── monitoring/
├── load_test/
└── global/

state는 versioning과 public access block, 비암호화 전송 거부 정책을 둔 S3 backend에 환경별 key로 분리하고 S3 native lockfile을 사용했습니다.

backend "s3" {
  bucket       = "…-tfstate"
  key          = "env/prod/terraform.tfstate"
  use_lockfile = true
  encrypt      = true
}

이 구조에서는 monitoring 변경이 prod state를 잠그지 않고, load_test 환경은 독립적으로 생성·삭제할 수 있습니다. 여기의 use_lockfile은 state 동시 수정 방지용 .tflock이며 provider 버전을 고정하는 .terraform.lock.hcl과는 다른 파일입니다. 현재 저장소는 provider dependency lock file을 커밋하지 않아 plan과 apply의 provider 재현성은 별도 과제로 남아 있습니다.

모듈 경계와 현재 코드화 범위

환경 디렉터리가 “어디에 적용하는가”를 표현한다면 모듈은 “무엇이 함께 바뀌는가”를 표현하도록 나눴습니다.

  • app_stack — prod·stage가 공유하는 EC2, RDS, Security Group과 서버 초기화
  • monitoring_stack — 모니터링용 EC2·Security Group, Docker 설치와 Nginx 설정 스크립트 기반
  • shared_resources — 환경을 가로질러 공유하는 S3, CloudFront, Lambda, ACM

prod와 stage는 같은 app_stack을 사용하되 인스턴스 크기, ingress 규칙, 도메인과 RDS 설정을 입력값으로 분리했습니다. 복사된 리소스 정의가 서로 다른 방향으로 변하는 drift를 줄이기 위한 구조입니다.

monitoring_stack이라는 이름만으로 관측 스택 전체가 코드화됐다고 볼 수는 없습니다. 현재 Terraform은 host와 network 기반을 만들지만 Grafana·Loki·Prometheus 컨테이너 정의, datasource·dashboard·alert, 인증, 로그·메트릭 보존 기간과 volume backup은 포함하지 않습니다. Nginx 설정도 생성된 스크립트를 서버에서 수동 실행하는 범위입니다.

공개 PR로 확인한 실제 전환 순서

공개 PR 이력에서 먼저 확인되는 것은 prod·stage Terraform 기반 PR #4이고, 그다음 S3·Lambda 공유 리소스 PR #13, 이후 모니터링 기반 PR #14입니다. 따라서 “모니터링에서 먼저 검증한 뒤 prod·stage로 확산했다”는 순서로 설명하는 것은 실제 이력과 맞지 않습니다.

전환은 환경별 root와 공통 module을 나누며 범위를 넓힌 과정이었지만, 모든 기존 리소스를 같은 수준으로 import하거나 모든 서버 내부 설정을 선언적으로 관리한 것은 아닙니다.

app_stack에서는 Docker 같은 최초 부팅 작업을 cloud-init으로 처리하고, 일부 Nginx·관측 에이전트 템플릿 hash 변경을 null_resource와 SSM RunCommand로 반영합니다. 이 흐름은 monitoring root에 동일하게 적용되지 않습니다. 모니터링 쪽에는 별도의 수동 Nginx 설정 스크립트가 남아 있으므로 두 구현 범위를 구분해야 합니다.

IaC 이후에도 남은 재현성과 운영 한계

Terraform 도입으로 환경 경계와 변경 내용을 코드에서 확인할 수 있게 됐지만, 현재 저장소만으로 “검토한 plan과 동일한 인프라가 적용되고 모든 설정과 secret이 재현된다”고 말할 수는 없습니다.

  • provider 고정 — .terraform.lock.hcl을 ignore하고 provider constraint가 >= 5.0인 상태라 실행 시점에 선택되는 provider가 달라질 수 있습니다. dependency lock file을 검토·커밋해야 합니다.
  • 검토한 plan의 동일 적용 — PR에서는 speculative plan을 만들지만 main workflow는 새 terraform apply -auto-approve를 실행합니다. 승인한 saved plan artifact를 그대로 apply하는 구조가 아니며, plan artifact에는 민감한 인프라 정보가 포함될 수 있어 접근과 보존도 제한해야 합니다.
  • secret 관리 — 현재 infra 저장소에는 private config/secrets submodule과 .tfvars 소비가 남아 있습니다. Parameter Store 입력 전환 PR #29는 병합되지 않았으므로 “infra secret을 모두 Parameter Store로 전환했다”고 표현할 수 없습니다.
  • drift 가시성 — ignore_changes가 user data, AMI와 key name의 변경을 숨깁니다. 의도적으로 책임을 분리한 값과 놓치면 안 되는 drift를 문서화해야 합니다.
  • 명령형 경계 — null_resource와 local-exec로 SSM 명령을 호출하면 부분 실패와 재실행 상태가 Terraform resource lifecycle에 완전히 표현되지 않습니다. provisioner는 마지막 수단으로 제한하고 timeout·idempotency·실패 복구를 별도로 검증해야 합니다.
  • network와 host — default VPC, public IP와 SSH key가 남고 일부 보조 스크립트는 chmod 777과 host networking을 사용합니다. 최소 권한과 private 접근 기준으로 좁힐 범위입니다.
  • 인증서와 관측 데이터 — standalone Certbot 갱신 시 80 port 점유, cron 실행 권한과 중복 등록을 dry-run으로 확인해야 합니다. dashboard·alert·retention·backup·volume 복구 절차도 아직 Terraform 관리 범위가 아닙니다.
  • backend bootstrap — state bucket의 versioning·암호화·public 차단은 구현됐지만, 그 bucket을 자신이 제공하는 backend로 전환하는 bootstrap/import 절차는 문서화가 더 필요합니다.

현재 확인할 수 있는 성과는 환경별 root와 state, 공통 module, 일부 host·network·배포 기반을 코드와 PR에서 추적할 수 있게 만든 것입니다. 다음 단계는 provider lock과 saved plan 동일 적용, secret 전환, 모니터링 구성·데이터 수명주기의 코드화로 그 재현성 범위를 넓히는 일입니다.

현재 구현 근거: monitoring root · Terraform Plan workflow · Terraform Apply workflow. 이 링크들은 현재 root와 plan·apply 흐름의 근거이며, 관측 도구 설정 전체나 saved plan 동일 적용이 완료됐다는 근거는 아닙니다.
