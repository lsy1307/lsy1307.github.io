Consistency · S3

S3 보상 삭제를 구현하며 놓친 비동기 경계

이미지 업로드와 대학 정보 저장을 한 유스케이스로 묶었던 당시 구현을 돌아보며, @Async void 때문에 호출부의 보상 삭제가 완료·실패를 보장하지 못했던 이유를 정리했습니다.

하나의 요청과 두 개의 저장소

관리자가 대학을 생성할 때 로고와 배경 이미지를 먼저 S3에 업로드하고, 그 URL을 대학 정보와 함께 DB에 저장합니다. 사용자에게는 하나의 작업이지만 S3와 RDB는 같은 트랜잭션에 참여하지 않습니다.

로고 업로드는 성공했는데 배경 업로드나 DB 저장이 실패하면 DB는 롤백되어도 S3 객체는 남습니다. 반대로 클라이언트가 이미지 업로드와 대학 생성 API를 따로 호출하면 실패 순서와 정리 책임까지 클라이언트가 알아야 했습니다.

핵심 경계@Transactional은 선택된 DB 트랜잭션의 변경만 되돌립니다. S3 객체는 같은 원자적 커밋에 참여하지 않으며, 애플리케이션 보상 로직도 별도의 실패 가능성을 갖습니다.

대학 생성 유스케이스로 업로드 책임 통합

대학 생성·수정 API가 JSON과 이미지 파일을 multipart/form-data로 함께 받도록 변경했습니다. 클라이언트가 이미지 URL을 직접 전달하는 필드는 제거하고, 서버가 파일 검증·업로드·URL 저장을 한 흐름에서 관리합니다.

  • 생성: 로고와 배경 이미지를 필수로 업로드한 뒤 대학 정보 저장
  • 수정: 전달된 이미지만 새로 업로드하고, 파일이 없으면 기존 URL 유지
  • 서버가 생성한 URL만 DB에 저장해 임의 URL과 잘못된 경로 유입 방지

서비스의 결합도는 조금 커지지만 이미지 업로드가 대학 관리에 종속된 작업이라는 사실을 코드에 드러내는 쪽을 선택했습니다. 다만 유스케이스를 한 메서드로 모았다는 사실이 S3와 DB의 원자성을 만들어 주지는 않습니다.

당시 구현의 실제 한계: @Async void

당시 S3 업로드·삭제 메서드는 @Async와 void 반환형을 사용했습니다. 호출자는 비동기 작업이 실행기에 제출되면 다음 단계로 진행하며, 작업 스레드에서 나중에 발생한 업로드 예외를 같은 try-catch로 받을 수 없습니다.

@Async
public void upload(/* ... */) {
    // 호출부가 반환된 뒤 별도 스레드에서 실행될 수 있음
}

try {
    upload(logoFile);       // 완료 결과를 받지 못함
    repository.save(entity);
} catch (RuntimeException e) {
    // 비동기 작업 스레드의 예외는 여기로 전파되지 않음
}

AsyncUncaughtExceptionHandler 등으로 예외를 기록할 수는 있지만, 그 예외가 원래 호출의 DB 트랜잭션에 다시 참여하거나 DB를 롤백시키지는 않습니다. 작업 스레드에 호출 스레드의 트랜잭션 경계가 그대로 이어진다고 가정할 수도 없습니다. 삭제 역시 @Async void라면 호출부는 정리가 완료됐는지 확인할 수 없습니다.

정정당시 구현은 업로드 책임과 삭제 시도를 서버에 모은 구현입니다. “업로드 성공 후 DB 실패 시 S3 객체가 반드시 삭제된다”거나 “원래 예외를 항상 보존하면서 보상한다”는 보장은 성립하지 않습니다.

구현 근거: solid-connect-server PR #782에서 대학 생성·수정 API 통합과 보상 삭제 시도를, PR 당시의 FileUploadService에서 @Async void 업로드 경계를 확인할 수 있습니다. 두 링크는 당시 구현 범위와 한계를 보여 주는 근거이며, S3와 DB의 원자성이나 보상 성공을 증명하지 않습니다.

동기 완료를 확인할 수 있을 때의 재설계안

아래 흐름은 당시 @Async void 구현을 설명하는 코드가 아니라, 업로드가 동기적으로 완료되거나 CompletableFuture<UploadedFile> 같은 결과를 호출자가 기다릴 수 있도록 바꿨을 때 가능한 재설계안입니다. 이 조건에서만 생성된 객체의 key를 기록하고 이후 동기 예외에 대해 역순 삭제를 시도할 수 있습니다.

// REDESIGN PROPOSAL — 당시 구현 아님
UploadedFile logo = null;
UploadedFile background = null;

try {
    logo = uploadAndWait(logoFile);
    background = uploadAndWait(backgroundFile);
    return repository.saveAndFlush(university);
} catch (RuntimeException original) {
    deleteQuietly(background);
    deleteQuietly(logo);
    throw original;
}

saveAndFlush는 SQL 실행 시 드러나는 일부 DB 제약 위반을 catch 범위 안에서 확인하는 데 도움이 되지만, 트랜잭션 커밋의 모든 실패를 앞당기지는 못합니다. 또한 S3 작업의 완료를 기다리지 않는 당시 비동기 구조의 문제도 해결하지 않습니다.

이 재설계도 분산 트랜잭션은 아닙니다. 삭제 요청 자체가 실패하거나 프로세스가 종료될 수 있으므로 보상은 best effort입니다. 더 강한 복구가 필요하다면 업로드 상태를 DB에 기록하고, 재시도 가능한 작업과 고아 객체 정리 작업을 별도로 운영해야 합니다.

공개 URL과 삭제 키 분리

리사이즈 대상 이미지는 DB와 클라이언트에 전달하는 URL이 실제 원본 업로드 키와 다를 수 있습니다. 호출부가 URL에서 S3 key를 역산하면 저장 정책이 바뀔 때마다 함께 수정되어야 합니다.

업로드 결과가 fileUrl과 내부용 deletionKey를 함께 갖도록 한 구조는 삭제 대상을 정확히 지정하는 데 유용합니다. deletionKey는 JSON 응답에서 숨기고 S3 서비스 내부에서 사용합니다. 다만 key를 안다는 사실과 비동기 삭제의 완료를 확인했다는 사실은 다릅니다.

record UploadedFile(
    String fileUrl,
    @JsonIgnore String deletionKey
) {}

표시용 식별자와 작업용 식별자를 나누면서 호출부가 S3 경로 정책을 역산하지 않게 했습니다. 이는 경로 책임을 분리한 개선이지, 보상 실행의 성공을 보장하는 장치는 아닙니다.

읽기 좋은 경로와 유일한 경로의 균형

대학 영문명을 소문자·공백 치환·특수문자 제거 규칙으로 slug화하면 읽기 좋은 디렉터리를 만들 수 있습니다. 하지만 정규화는 손실 변환이므로 서로 다른 이름이 같은 slug가 될 수 있고, 영문명 자체도 유일하지 않을 수 있습니다.

디렉터리 prefix는 영문 slug를 유지하되, unique 제약이 있는 한글명의 해시를 suffix로 사용했습니다. 사람이 알아보기 쉬운 경로와 충돌 회피를 함께 얻기 위한 선택입니다.

directory = slug(englishName)
          + "_"
          + shortHash(uniqueKoreanName);

결론: 보상 의도와 보상 보장은 다르다

당시 구조에서 비동기 예외는 호출부의 원래 예외 흐름과 분리됩니다. 따라서 보상 삭제 실패가 원래 예외를 덮지 않는다고 보장할 수도, 남은 객체를 자동으로 식별해 정리한다고 말할 수도 없습니다. 확실히 말할 수 있는 범위는 서버가 업로드·삭제 책임을 소유하도록 API 경계를 바꿨고, 실패 시 정리를 시도하는 코드를 두었다는 점까지입니다.

  • 기존 이미지 교체 후 old image 삭제는 DB 커밋 이후 처리해야 하므로 별도 과제로 분리
  • 이름 변경만으로 기존 S3 객체를 이동하지 않음
  • @Async void 업로드·삭제의 완료와 예외를 호출부에서 확인할 수 없음
  • DB 커밋 실패, 비동기 작업 실패, 프로세스 종료 뒤 orphan 객체가 남을 수 있음
  • 장기적으로 상태 기록, 멱등 재시도, orphan 객체 탐지·정리 정책이 필요

이번 회고의 결론은 보상 코드를 두는 것과 복구를 보장하는 것이 다르다는 점입니다. 강한 표현보다 실제 실행 모델을 먼저 확인하고, 동기 완료 확인 또는 지속 가능한 재처리 구조가 없는 한 best effort로 설명해야 합니다.
