환경 전환 20분을 3분으로: 설정 외부화와 자동화

분산된 설정과 secret을 Parameter Store로 옮기고, dev와 loadtest 환경 전환을 반복 가능한 작업으로 만든 과정입니다.

환경을 바꿀 때마다 사람이 설정을 맞췄다

기존 설정은 application.yml과 Git 서브모듈에 흩어져 있었습니다. dev, prod, loadtest를 오갈 때 DB endpoint와 profile 설정을 직접 확인해야 했고, 설정 충돌과 누락 위험이 반복됐습니다.

특히 부하 테스트 환경은 자주 켜고 끄기 때문에 매번 20분 가까운 준비 시간이 들었습니다. 테스트 자체보다 환경을 맞추는 일이 더 큰 비용이 되는 구간이었습니다.

목표설정의 저장 위치를 하나로 모으고, 실행 환경은 “값을 들고 있는 주체”가 아니라 “필요한 값을 읽는 주체”로 바꿉니다.

Parameter Store에 환경별 경계를 만들었다

설정은 /solid-connection/{local|dev|prod|loadtest} 경로로 분리했습니다. 애플리케이션은 Spring profile에 따라 필요한 경로만 읽습니다.

spring:
  config:
    import: "aws-parameterstore:/solid-connection/${spring.profiles.active}/"

# 구조 예시
/solid-connection/dev/db-endpoint
/solid-connection/prod/db-endpoint
/solid-connection/loadtest/db-endpoint

이 구조로 DB endpoint, secret, profile별 설정을 코드에서 분리했습니다. 설정 파일 변경과 애플리케이션 배포를 항상 묶지 않아도 되고, secret 서브모듈을 관리할 필요도 사라졌습니다.

환경 전환을 GitHub Actions의 입력값으로

환경 전환과 부하 테스트 리소스 생성을 스크립트로 묶고 GitHub Actions에서 실행하도록 구성했습니다. 사람이 여러 명령을 순서대로 입력하는 대신, 대상 환경을 선택하면 필요한 단계가 같은 순서로 반복됩니다.

자동화의 핵심은 명령을 짧게 만드는 데 있지 않았습니다. 전환 절차를 코드로 고정해 누가 실행해도 같은 결과를 얻는 것이 더 중요한 가치였습니다.

Access key 대신 IAM Role

EC2의 IAM Role에 Parameter Store 읽기 권한을 부여했습니다. 애플리케이션 코드나 인스턴스 내부에 장기 access key를 저장하지 않고도 실행 시점에 설정을 가져올 수 있습니다. 권한 범위는 서비스가 사용하는 경로로 제한했습니다.

  • 실행 환경에 credential 파일을 두지 않음
  • 서비스별 Parameter Store 경로로 읽기 범위 제한
  • 환경별 설정 변경 이력을 AWS와 배포 흐름에서 추적

결과: 테스트를 시작하는 비용을 낮췄다

BEFORE약 20분
AFTER2~3분

전환 시간은 약 85% 줄었습니다. 더 중요한 변화는 부하 테스트가 특별한 이벤트가 아니라 필요할 때 반복할 수 있는 작업이 됐다는 점입니다. 설정 외부화와 자동화가 함께 적용되어야 이 효과가 완성된다는 것도 배웠습니다.
