문제: 편리한 참조가 만든 넓은 변경 범위

Solid Connection의 지원 현황 조회는 지원 기간에 트래픽이 집중될 가능성이 큰 API입니다. 기존 Application 도메인은 대학, 학기, 사용자 등 여러 엔티티를 @ManyToOne으로 직접 참조했습니다.

초기에는 자연스러운 모델링이었지만 기능이 늘면서 두 문제가 함께 드러났습니다. 조회 경로에서는 지망별 다중 조회가 반복됐고, 설계 측면에서는 한 도메인의 수정이 Entity, Repository, Service, Test로 넓게 전파됐습니다. 연관관계가 있다는 사실만으로 N+1이 발생하는 것은 아니므로, 성능 원인은 실제 실행된 조회 흐름과 SQL을 기준으로 따로 확인했습니다.

개선 전략반복 조회는 일괄 조회로 줄이고, 객체 직접 참조는 ID 기반으로 전환했습니다. 성능 병목과 도메인 결합 문제를 각각 해결하면서 기존 API 응답 형식은 유지했습니다.

k6 기반 병목 확인

리팩터링 전에 지원 현황 API를 대상으로 k6 부하 테스트를 진행했습니다. 평균값만으로는 일부 느린 요청을 가릴 수 있어 p99를 함께 확인했고, 당시 기록한 기존 구조의 p99 응답 시간은 972ms였습니다.

실행 흐름을 추적했을 때 지망별로 반복되는 다중 조회가 주요 개선 지점이었습니다. 따라서 단순히 캐시를 추가하기보다 필요한 데이터를 한 번에 가져오도록 읽기 경로를 줄이는 것이 먼저라고 판단했습니다.

fetch join 하나로 끝내지 않은 이유

fetch join은 특정 조회에서 연관 엔티티를 함께 읽어 N+1을 줄이는 유효한 선택입니다. 다만 당시 병목에는 서비스 계층에서 지망별 조회를 반복하는 흐름이 포함돼 있었고, fetch join만 추가해도 도메인 객체의 직접 의존과 넓은 변경 범위는 남습니다. 그래서 fetch join을 배제한 것이 아니라, 조회 호출 구조와 도메인 참조 구조를 별도 과제로 다뤘습니다.

  • 엔티티 간 직접 참조를 Long id 참조로 전환
  • Repository가 화면 응답에 필요한 식별자를 명시적으로 일괄 조회
  • Service는 조회 결과를 지원 가능·지원 완료·삭제 제외 상태로 분류
  • 기존 DTO 형식은 유지해 프론트엔드 영향 최소화

조회와 분류 책임 분리

성능 측면에서는 지망별 개별 조회를 하나의 일괄 조회로 합치고, 서비스 계층에서 필요한 상태로 분류했습니다. 설계 측면에서는 엔티티 간 객체 참조를 식별자 참조로 바꿔 다른 엔티티의 생명주기에 대한 의존을 줄였습니다. 두 변경은 같은 리팩터링에서 진행했지만 목적과 효과는 구분했습니다.

// 개념 예시: 엔티티 자체가 아니라 식별자로 관계를 표현
class Application {
    private Long siteUserId;
    private Long universityId;
    private Long semesterId;
}

// 필요한 데이터를 일괄 조회한 뒤 응답 목적에 맞게 분류
applications.stream()
    .filter(applicationPolicy::isVisible)
    .collect(groupingBy(Application::getPreference));

리팩터링 과정에서는 엔티티 생성자, Repository, Service, Test를 함께 동기화했습니다. 응답 계약을 유지했기 때문에 프론트엔드는 별도 수정 없이 새 구조를 사용할 수 있었습니다.

구현 내용은 solid-connect-server PR #330에서 확인할 수 있습니다. @ManyToOne 직접 참조를 ID로 전환하고, 대학 ID 목록을 일괄 조회한 뒤 서비스 계층에서 응답 목적에 맞게 분류했습니다.

개선 결과

RECORDED P99972 → 366ms
RECORDED CHANGE약 62% 감소
QUERY PATH반복 조회 → 일괄 조회
API CONTRACT기존 DTO 유지

k6로 변경 전후를 비교한 결과 p99 응답 시간은 972ms에서 366ms로 약 62% 줄었습니다. 반복 조회를 일괄 조회로 바꿔 읽기 경로를 단순화했고, ID 참조 전환으로 도메인 사이의 의존과 수정 범위도 함께 낮췄습니다.

이 경험을 통해 성능 개선에서는 실행 SQL과 호출 횟수를 확인하고, 설계 개선에서는 의존 방향과 변경 범위를 함께 살펴야 한다는 기준을 세웠습니다.
